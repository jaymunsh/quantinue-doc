# 자동 다중 후보 파이프라인 작업 계획

## 요약

**제공 결과:** 티커 입력 없이 한 번 실행하면 공개 유니버스 50개를 선정하고, 실제 일봉 기술 분석 20개를 거쳐, 최종 후보 20개 각각에 05~11 역할을 실행한다. 운영실에서는 후보별 순위·진행률·판단·전체 상세 기록을 확인한다.

**핵심 결정:** 01~04의 공통 발견 작업은 한 번만 수행하고, 05~11만 후보별 독립 실행으로 분기한다. 후보 실패는 다른 후보를 막지 않으며 후보 순위와 실행 계보를 영속화한다.

**제외 범위:** 스케줄러 자동 반복, Alpaca 실주문, 다계정 분배는 포함하지 않는다. 호스트 5432의 어떤 자원도 사용하거나 검사하지 않는다.

**규모:** 중간
**위험:** 중간 — 공통 발견 컨텍스트를 후보별 실행으로 안전하게 재결합해야 한다.

## 범위

### 필수

- 기본 폼에서 티커 입력 제거 및 50 → 20 → 20 자동 실행
- 자동 실행에서 기본 티커를 강제로 포함하지 않음
- 후보 20개 순위대로 05~11 실행, 후보별 장애 격리
- 공통 01~04 결과와 근거 ID를 후보 실행 ID로 일관되게 재결합 및 체크포인트
- 운영실 후보 카드에서 05~11 상세 데이터 전체 노출, 데스크톱·모바일 무가로 스크롤
- 공개 SEC ticker→CIK 정식 해석과 앱 수명 캐시

### 금지

- 기존 단일 티커 API 계약 제거
- 실제 주문 안전 잠금 완화
- 비밀값 생성·기록·출력
- 관련 없는 dirty/untracked 파일 변경 또는 되돌리기

## 구현 및 검증

1. 계약과 저장소: `automatic`, `candidate_rank`, 후보 seed 체크포인트를 메모리/PostgreSQL 저장 경계에 추가한다. 후보 ID 재결합 불변조건을 테스트한다.
2. 발견 단계: 자동 요청에서는 숨은 요청 티커 편향 없이 50개를 선정하고 20개의 사용 가능한 일봉 분석을 완성하며 20개 후보를 결정론적으로 순위화한다.
3. 후보 단계: 01~04를 한 번 실행한 뒤 후보별 05~11을 순차 실행하고, 개별 실패를 기록한 뒤 다음 후보를 계속한다.
4. 공개 공시: NASDAQ ticker를 SEC CIK로 해석하고 검증하며 회사 ticker 인덱스를 앱 수명 동안 한 번만 가져온다.
5. 운영실: 후보 순위·상태·진행률·판단·확신도와 05~11의 사실/항목을 제한 없이 접어서 표시한다.
6. 문서/증거: README를 50 → 20 → 20으로 맞추고 런타임 감사 및 append-only ledger에 비밀값 없이 결과를 기록한다.

## 성공 기준

- 화면·문서·런타임이 모두 50 → 20 → 20으로 일치한다.
- 자동 실행에서 NVDA 등 숨은 기본 티커가 후보를 밀어내지 않는다.
- 후보별 순위와 계보가 재시작 가능한 저장소 체크포인트에 남는다.
- 20개 후보의 05~11 상세 결과가 가독성 있게 확인된다.
- 모든 최종 게이트와 독립 리뷰가 PASS한다.
