# Task 3: PostgreSQL app-order exposure reservation

## Result

Implementation is ready for independent verification. The PostgreSQL store now keeps the
app-owned reservation state in existing `tb_order` rows and never reads a broker account
balance, portfolio, or position.

## TDD evidence

The disposable PostgreSQL test was added before the implementation. It first failed because
the old store returned a boolean instead of the typed reservation result. It then exposed the
provider-status defect: a raw `accepted` value violated `tb_order`'s canonical status check.

The final durable contract is:

- Transaction locks are acquired in fixed order: account-wide capital cap, account/day quota,
  then order identity.
- The transaction validates an existing immutable reservation before replaying it, counts only
  `planned`, `submitted`, and `filled` reference notional across all trade dates, keeps daily
  attempts separate, and inserts only when both limits permit it.
- `failed` and `canceled` release planned exposure; terminal rows cannot be regressed by a
  later stale result.
- Role 10 normalizes raw provider status once before the context and store boundary. PostgreSQL
  domain reconciliation accepts only that canonical value and skips an already-terminal or
  already-equal durable state, so it does not perform a second accepted/submitted write.
