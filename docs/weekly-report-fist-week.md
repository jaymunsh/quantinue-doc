## 1. 주간 요약

1주차에는 공시·뉴스·기술지표·시장 데이터를 바탕으로 투자 후보를 분석하고, 전략 판단과 독립 비평, 리스크 계산, 모의 주문, 사후 리뷰 예약까지 연결하는 **01~11단계 투자 판단 파이프라인**을 구축했다.

설계 측면에서는 여러 문서에 흩어져 있던 데이터 구조와 처리 규칙을 하나의 통합 설계서로 정리하고, 테이블·필드·ENUM·시간대·에이전트 간 전달 규격을 통일했다. 구현 측면에서는 실제 공개 데이터와 PostgreSQL을 이용해 수집부터 판단·비평·리스크·모의 주문·리뷰 예약까지 이어지는 흐름을 확인했다.

투자 성향은 안정형·균형형·공격형을 모두 동시에 구현하지 않고 **공격형 1종만 1차 MVP 검증 대상으로 선정**했다. 공격형은 세 성향 중 매수 신호에 가장 민감해 전략 판단, 비평, 리스크 제한, 주문 차단과 같은 핵심 동작이 실제로 발생하는지 빠르게 확인하기에 적합하다. 또한 성향별 임계값과 계좌를 한꺼번에 운영할 경우 초기 검증 범위가 불필요하게 커지므로, 먼저 단일 성향·단일 계좌로 전체 파이프라인을 안정화한 뒤 안정형과 균형형으로 확장하는 방식을 선택했다.

이번 주의 핵심 성과는 단순히 투자 결과를 만드는 데 그치지 않고, 각 판단의 입력 데이터와 근거, AI 분석 결과, 코드 게이트, 실패 단계가 추적 가능한 형태로 남는 **감사 가능한 MVP 기반**을 마련한 것이다.

## 3. 주요 진행 내용

### 3.1 통합 설계 및 공통 계약 정립

- 기존 설계 문서와 수집·전략·비평 관련 문서를 통합해 1차 MVP 기준 설계서를 확정했다.
- 전체 시스템을 아래의 11개 역할로 구분하고, 앞 단계의 결과가 다음 단계로 전달되는 단일 파이프라인으로 정리했다.

  `01 유니버스 선정 → 02 기술 분석 → 03 일일 후보 선정 → 04 매크로 분석 → 05 공시 분석 → 06 뉴스 분석 → 07 전략 판단 → 08 비평 검증 → 09 리스크 계산 → 10 모의 주문 → 11 리뷰 예약`

- 전 테이블과 API가 같은 의미로 데이터를 해석하도록 다음 공통 규칙을 확정했다.

  - 점수 범위를 `0~1`로 통일
  - 불리언은 `BOOLEAN`과 `is_`/`has_` 접두사 사용
  - 상태·판정 값은 영문 소문자 ENUM 형태로 통일
  - 모든 시각은 `TIMESTAMPTZ(UTC)` 기준으로 저장
  - 행 생성 시각은 `created_at`, 사건 발생 시각은 `published_at` 또는 `filed_at`으로 분리
  - 에이전트 간 데이터는 DB 키와 구조화된 JSON 계약으로 전달

### 3.2 데이터 수집·분석 구조 구현

- 공시와 뉴스를 **원본 원장**과 **분석 집계 신호**의 2단 구조로 분리했다.

| 구분 | 원본 저장 | 집계 신호 | 목적 |
|---|---|---|---|
| 공시 | `tb_disclosure` | `tb_disclosure_signal` | 공시 원문 보존 및 종목별 영향 요약 |
| 뉴스 | `tb_news` | `tb_news_signal` | 기사 원문 보존 및 종목별 대표 신호 생성 |

- 원본 데이터는 수정하지 않는 append 방식으로 보존해 근거 역추적과 향후 백테스트가 가능하도록 했다.
- SEC 공시와 Google News RSS 기반의 공개 데이터를 수집하고, 로컬 LLM을 통해 중요도·감성·신뢰도·위험 요인을 구조화했다.
- 로컬 LLM은 상용 모델보다 분석 정확도와 추론 성능이 부족할 수 있다는 한계가 있으나, 1주차의 목표가 투자 판단 성능 검증보다 **수집 → 분석 → 전략 → 비평 → 리스크 → 모의 주문 → 리뷰로 이어지는 전체 흐름의 연결과 동작 확인**에 있었기 때문에 우선 적용했다. 또한 MVP2에서는 분석 대상과 실행 주기가 확대되면서 LLM 호출량과 비용이 증가할 가능성이 있으므로, 초기 단계부터 로컬 모델을 활용해 반복 실행 비용을 줄이고 향후 상용 모델 적용 범위와 예산 기준을 판단할 수 있는 기반을 마련하고자 했다.
- 저신뢰 출처 차단, 공시와 뉴스 간 교차 확인, 점수 범위 검증 등 데이터 오염 방지 규칙을 반영했다.

### 3.3 전략 판단과 독립 비평 구현

- 전략가가 기술·공시·뉴스·모델 점수를 종합해 `buy` 또는 `hold` 제안과 확신도를 생성하도록 구현했다.
- AI의 문장형 판단과 실제 매수 가능 여부를 분리하고, 최종 결정은 코드 정책 게이트가 담당하도록 구성했다.
- 비평가는 전략가의 제안을 그대로 수용하지 않고 다음 항목을 독립적으로 검증하도록 구현했다.

  - 매수 제안 및 필수 근거 존재 여부
  - 데이터 신선도와 출처 신뢰도
  - 공시·뉴스 간 근거 충돌
  - 과열, 거래정지 등 하드 규칙 위반
  - 모델 확신도와 실제 근거 수준의 일치 여부

- 비평 결과와 반대 근거를 `tb_critic_verdict`에 저장해 이후 리스크·주문 단계가 판단 이유를 추적할 수 있도록 했다.

### 3.4 리스크·모의 주문·리뷰 연결

- 비평을 통과한 신호에 대해서만 계좌 잔고와 리스크 규칙을 적용해 주문 수량, 진입가, 손절가, 익절가를 계산하도록 연결했다.
- 1차 MVP에서는 외부 실계좌로 주문을 보내지 않고 로컬 모의 체결 결과와 주문번호만 생성하도록 제한했다.
- 최종 판단 결과는 T+1~T+5 사후 평가 대상으로 등록했다.
- 실제 가격 스냅샷 저장과 최종 성과 평가는 운영자가 처리 API를 호출할 때 멱등하게 수행하도록 설계했다.

### 3.5 운영·시연 화면 구성

- 한 화면에서 다음 내용을 순서대로 확인할 수 있는 MVP 쇼케이스를 구성했다.

  - 최근 실행 환경과 파이프라인 실행 요약
  - 분석 대상별 01~11단계 진행 상태
  - 공시·뉴스 분석 결과 및 대표 근거
  - 전략 판단과 비평 결과
  - 리스크·주문·리뷰 상태
  - 단계별 처리 시간과 체크포인트
  - 과거 실행 및 실패 이력

- 새로고침 후에도 PostgreSQL에 저장된 분석 결과와 데이터 계보를 확인할 수 있어 운영·감사 관점의 추적성을 확보했다.

## 4. 구현 및 검증 결과

### 4.1 파이프라인 검증

| 검증 항목 | 확인 결과 |
|---|---|
| 공개 데이터 수집 | SEC 공시와 뉴스 데이터의 수집·원본 저장 확인 |
| 분석 결과 구조화 | 공시·뉴스의 점수, 근거, 출처 및 모델 정보를 구조화해 저장 |
| 전략·비평 분리 | 전략가의 제안과 비평가의 독립 판정을 별도 단계·테이블로 기록 |
| 코드 안전장치 | AI 판단과 별개로 신뢰도·임계값·리스크 규칙을 코드로 적용 |
| 데이터 계보 | 입력 원본에서 전략·비평 결과까지 참조 관계 추적 가능 |
| 주문 안전성 | 외부 실계좌 전송 없이 로컬 모의 주문으로 제한 |
| 리뷰 연결 | 최종 판단을 T+1~T+5 사후 평가 대상으로 등록 |
| 운영 가시성 | 단계별 상태·처리 결과·오류·체크포인트를 화면과 DB에서 확인 |

이번 실행 결과는 특정 종목의 투자 적합성이나 수익률을 검증하기 위한 것이 아니라, **데이터가 정해진 계약에 따라 이동하고 안전장치를 거쳐 최종 결과와 근거로 남는지 확인하기 위한 기능 검증**으로 해석한다.

### 4.2 설계 협의 결과

- 사이클의 계획 시각, 실제 저장 시각, 판단에 사용한 원본 시각을 서로 다른 의미로 분리했다.
- 공시·뉴스 원본과 집계 신호의 소비자를 역추적해 사용되지 않는 필드를 제거하고 필요한 집계 규칙을 보강했다.
- 전략 담당자가 점수만이 아니라 점수별 산출 근거를 확인할 수 있도록 `reason` 구조를 확장했다.
- 공시도 뉴스와 동일하게 건수와 상위 근거를 전달하도록 계약을 보완했다.
- 문서 합의가 코드보다 먼저 확정되고, 스키마·타입·설정 정본을 먼저 반영한 뒤 담당별 구현을 교체하는 순서를 합의했다.

| **리뷰·회고 담당** | 리뷰·회고 | 판단 결과의 T+1~T+5 평가 구조, 학습용 리뷰 데이터 계약 정리 | `tb_review_price_snapshots`, `tb_review`, 회고·학습 기준 |

### 협업 원칙

- 각 담당자는 자신이 생성하는 데이터의 정확성과 스키마 준수를 1차 책임으로 갖는다.
- 다음 단계 담당자는 입력 데이터의 누락·범위·ENUM·시간 기준을 소비자 관점에서 검증한다.
- 공통 계약 변경 시 설계 문서와 코드의 계약 정의를 함께 갱신한다.
- 투자 판단은 AI 단독 결과로 확정하지 않고, 비평과 코드 게이트를 반드시 거친다.

### 공통 후속 조치

1. 문서에서 합의한 계약을 `schema.sql`, Pydantic 스키마·ontology, `pipeline.yaml`의 정본 3영역에 먼저 반영한다.
2. 이후 수집·전략·비평·리뷰 담당 코드와 픽스처를 새 계약으로 교체한다.
3. 전 담당 교체가 끝나기 전에는 구 계약과 신 계약을 섞어 실행하지 않는다.
4. 통합 시나리오를 재실행해 멱등성, 데이터 계보, 점수별 근거 전달을 확인한다.

## 7. 리스크 및 유의사항

- 이번 결과는 **투자 수익성을 증명한 것이 아니라 시스템의 처리 흐름과 안전장치를 검증한 결과**다.
- 로컬 LLM의 판단 결과는 모델 성능을 대표하는 최종 품질 지표로 보지 않는다. 이번 단계에서는 실행 편의성과 MVP2의 잠재적인 LLM 호출 비용을 함께 고려해 전체 파이프라인의 정상 연결 여부를 확인하는 용도로 사용했다. 이후에는 실제 호출량과 비용을 측정한 뒤, 판단 품질이 중요한 단계에는 상용 모델을 적용하고 반복적인 수집·분석 단계에는 로컬 또는 경량 모델을 사용하는 방식으로 모델 구성과 예산을 검토한다.
- 11/11단계 완료는 T+5 성과 평가 완료가 아니라 **리뷰 대상 등록 완료**를 의미한다.
- 실제 주문은 발생하지 않았으며, 주문 단계는 로컬 모의 처리로 검증했다.
- 1차 뉴스 계약은 제목과 RSS 스니펫만 사용하므로, 본문 수준의 상세 해석에는 한계가 있다.
- 문서 계약과 코드 반영 시점이 어긋나면 데이터 생산자와 소비자가 서로 다른 스키마를 사용할 수 있으므로 정본 우선 적용 순서를 지켜야 한다.
- 실거래 전환 전에는 시크릿 관리, DB 권한 분리, 킬스위치, 프롬프트 인젝션 방어, 장기 운영 테스트가 추가로 필요하다.

## 8. 차주 계획

MVP2 계획에서는 1차의 단일 01~11 선형 실행을 그대로 확장하기보다, 실패 범위를 줄이고 각 기능을 독립적으로 운영할 수 있는 **잡(JOB) 기반 구조**로 전환하는 것을 핵심 방향으로 삼는다.

1. **스케줄·멱등성 강화**<br>
   거래일 캘린더와 계획 슬롯을 기준으로 각 작업이 자동 실행되도록 하고, 같은 슬롯의 중복 실행과 재시작 이후 누락을 방지한다.

2. **계약 및 분석 범위 확장**<br>
   #19·#21·#23에서 합의한 시각·계보·점수별 근거 계약을 코드에 동기화하고, 유니버스 → 기술 분석 → 일일 후보의 선별 구조를 설정값 중심으로 확장한다.

3. **판단 방어선과 계좌 정책 보강**<br>
   출처 신뢰도, 강한 악재, 매크로 위험, 투자 성향별 문턱을 코드 정책으로 분리하고 다중 계좌·서킷브레이커 기반을 마련한다.

4. **매수 이후 청산·평가 흐름 구축**<br>
   시스템이 매수뿐 아니라 손절·익절·보유기간·하드 이벤트에 따라 청산할 수 있도록 하고, 현금과 보유 자산을 합산한 계좌 시가평가를 구현한다.

5. **리뷰·학습 및 운영 가드 자동화**<br>
   T+5 결과 채점, 리뷰 lesson의 다음 판단 재사용, 벤치마크 비교, LLM 사용량·비용 기록, 예산 초과 차단, 소스 상태·실패 알림을 연결한다.

6. **운영 화면과 배포 기반 준비**<br>
   관리자용 스케줄·계좌·비용·재실행 화면과 사용자용 읽기 전용 성과·거래·회고 화면을 정리하고, 로컬 원클릭 기동과 DB 백업을 우선 완성한다.

## 9. 주간 결론

1주차에는 Quantinue의 핵심 목표인 **“공개 데이터를 수집해 투자 판단을 만들고, 독립적으로 반박한 뒤, 코드 기반 리스크 통제를 거쳐 모의 주문과 회고로 연결하는 흐름”**을 실행 가능한 형태로 구현했다.

실제 데이터 기반 실행을 통해 근거가 충분하지 않은 상황에서 주문을 생성하지 않는 안전 동작을 확인했다. 동시에 팀 협의에서는 단순 실행 성공 여부보다 재실행 멱등성, 입력 데이터 계보, 공시·뉴스 집계 기준, 점수별 판단 근거가 장기적으로 더 중요한 품질 기준이라는 점을 확인했다.

따라서 현재 상태는 **“1차 MVP의 핵심 구조와 관찰 가능성을 확보했고, 담당자 간 데이터 계약도 주요 쟁점에 대한 합의를 마쳤으며, 다음 단계에서는 이를 잡 기반 자동 운영·청산·학습 구조로 확장해야 하는 수준”**으로 판단한다.
