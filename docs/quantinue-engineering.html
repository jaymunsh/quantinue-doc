Boot & Config

기동 경로와 설정 체계

앱이 뜨는 순서 — main.py::create_app

  • Settings()(환경변수) → 로깅 → load_mvp2_config(config/pipeline.yaml)(정책)
  • 스토어 선택: database_mode로 memory / postgres 분기. 메모리 스토어면 JOB 러너가 안 뜬다 — tb_job_run(멱등의 근거)이 없기 때문
  • 시장 데이터 어댑터는 한 번만 생성해 유니버스·매크로 JOB이 공유 (예전엔 두 번 만들어 하나가 안 닫혔다)
  • build_job_runner(...) — 자격증명 유무로 수집 JOB 등록이 갈리고, 판단 JOB은 언제나 선다
  • FastAPI lifespan에서 task_group.start_soon(job_runner.run_forever) — 단, mvp2.jobs.enabled=true일 때만. 종료는 cancel_scope(결함 18의 수리 지점)
  • run_forever: 60초마다 tick — 뉴욕 세션일이 아니면 전 JOB "holiday" 스킵, 세션일이면 due인 JOB만 순서대로

함정 목록 — 전부 실제로 밟은 것

  • -p no:unraisableexception 없이 통합을 돌리면 15건이 가짜 실패 — asyncpg GC의 ResourceWarning이 에러로 승격된다. 로직 실패가 아니다.
  • 통합은 반드시 새 컨테이너에서 — 같은 컨테이너 재실행은 멱등 가드가 옛 행을 지켜내 실패를 가린다. 문서화된 함정을 실제로 밟았다.
  • 실 스모크 재측정 전에 슬롯을 비울 것 — save_signal의 on_conflict_do_nothing이 이미 성공한 슬롯의 첫 실행 행을 읽게 만든다. tb_job_run의 해당 슬롯과 그 행을 먼저 삭제.
  • Alpaca 키 재발급 후 앱 재시작 필수 — 기동 중 앱은 옛 키를 메모리에 들고 있다.
  • 주말 스모크는 평일 결함을 못 본다 — 결함 10(FK 위반)은 구 러너와 같은 trade_date를 쓰는 평일에만 터졌다.
  • mock LLM의 고정값은 갈래를 은폐한다 — reject 갈래(결함 11)·성향 격차는 실 LLM에서만 드러난다.
  • 테스트가 반환 id만 보면 유령을 통과시킨다 — 결함 19. 저장 후 컬럼을 실제로 읽어 대조할 것.
  • 관제실 숫자는 원장과 대조해서 믿는다 — 결함 15·16. 화면 카운트와 tb_job_run.detail 문자열 일치가 검증 기준.
  • 프로브가 남긴 행은 주문 참조 0을 확인하고 삭제 — 실험이 원장에 남으면 다음 측정이 오염된다.
  • 신규 설치 == 마이그레이션 — 스키마 4곳 미러 · 제약 정의 문자열 일치 · 마이그레이션 2회 멱등 · 카탈로그 개수 대조(제약 159 · 인덱스 48).
  • 실 API는 추정 금지 — 한 요청 400종목까지 OK, 심볼 구분자는 점(BRK.B), 쓰레기 심볼 하나가 배치 전체를 400으로 죽임, 뉴스 limit 상한 50. 전부 실측으로만 알 수 있었다.
