2차 확정 사항

R1

2차 = AI 페이퍼 펀드 서비스

사람 손 없이 도는 무인 운영 위에 두 개의 화면을 얹는다 — 관리자(펀드매니저 ERP: 계좌 CRUD·운영·비용·리포트)와 유저(조회 전용 포털). 리서치 신호(01~08)는 사이클당 1회 공용 생성, 계좌는 성향별 파라미터로 구독·실행(09~10)하므로 LLM 비용이 계좌 수와 무관하게 고정된다. 실계좌 전환은 범위 외(페이퍼 전용).

R2

앱 내장 스케줄러 · cycle_ts 슬롯 멱등

FastAPI 앱 안의 백그라운드 루프가 역할별 실행 창(뉴욕 시간)에 파이프라인을 자동 구동한다. cycle_ts는 실행 시각이 아니라 주기 슬롯 경계로 내림한 값이라, 같은 슬롯을 몇 번 실행해도 UNIQUE(ticker, cycle_ts)가 중복을 막는다(멱등). 슬롯은 계산값이라 마스터 테이블을 두지 않고, 실행 기록은 기존 pipeline_runs가 담당.

배포 로컬 → AWS주문 계열 개장 중만

판단 방어선 3종 전부 구현

R4

LLM 3단 실행 모드

mock(가짜 점수 · 단위 테스트) → 로컬 qwen(이 PC의 MLX · 개발·통합 테스트, 무료) → OpenAI gpt-4o-mini(운영). 모델 전환은 config 몇 줄. 단 점수 눈금은 모델마다 다르므로 로컬은 배관 검증용이고, 문턱값 캘리브레이션은 운영 모델 실측으로 한다.

R5

신호 품질 7건 (실무 개정 3건 포함)

reason를 점수별 근거 객체(JSONB)로 확장(#23) — 유저 화면 "거래 이유"의 원천. 대표 기사는 관련성 필터(오귀속 차단) 통과분 중 importance×신뢰무게 최상위로 개정. Form 4는 템플릿 조립하되 자발 매수(코드 P)만 유의미 신호로. consensus는 설명용 지표로만(게이트 아님 — 좁은 깔때기 방지). peak_importance 계산·halted 체크도 구현.

R6

학습 루프 · 비용 가드

T+5(영업일) 회고 채점을 자동화하고 lesson을 생성해, 07 전략이 같은 종목 최근 lesson을 프롬프트에 주입받는다 — "회고가 다음 판단을 개선한다"는 서사의 완성. LLM 호출을 전부 기록해 관리자 대시보드에 사용량·비용을 노출하고, 일일 예산 상한 초과 시 호출을 멈춘다(그 사이클 신규 매수 보류).

R7

실무 보강 6건

1차에 빠져 있던 실무 기본기 — ① 매도·보유 재평가(보유 종목 상시 분석 + 청산 3층: 가격·시간·논지 붕괴) ② 포트폴리오 서킷브레이커(일일 손실 한도 도달 시 당일 신규 매수 중단) ③ 텔레그램 알림 ④ NYSE 휴장일 캘린더 ⑤ 계좌 정합 대사 ⑥ SPY 벤치마크. ①이 07 파이프라인 구조에 손대는 최대 설계 작업.

R8

스크리닝 깔때기 원설계 복원

Conventions

공통 컨벤션 — 어휘·타입 규칙

이 8규칙이 문서 전체 어휘의 정본이다. 스키마·전달 JSON·코드 표기가 어긋나면 이 규칙이 이긴다. (1차에서 확정·계승, 규칙 4는 2차에서 개정.)

  1. Boolean은 BOOLEAN 타입 + is_/has_ 접두. INT 0/1 금지. 상태는 is_, 보유·존재는 has_ (is_hard_blocked·has_signal).
  2. ENUM은 영어 소문자 snake_case, 물리 타입은 TEXT. Postgres 네이티브 ENUM·참조 테이블 FK를 쓰지 않고, 허용값 검증 1차 책임은 백엔드(Pydantic Literal). event_type 허용값 정본 = ontology.py.
  3. 시각은 *_at = TIMESTAMPTZ(UTC 저장, 표시만 KST), 날짜는 *_date = DATE.
  4. 점수는 전부 0~1. DB는 NUMERIC + CHECK(0~1), 표기·전달은 소수 3자리. 2차 개정 — 1차의 "NUMERIC(4,3) 예외 없음"에서 물리 정밀도 강제는 해제(값 범위만 강제, 코드 현실 수용).
  5. 전달 JSON 필드명 = DB 컬럼명. 단계마다 이름을 바꾸지 않는다(개명 금지) — 추적·디버깅의 기준.
  6. 이름 규칙 — 원본 원장은 자연 명사(tb_news), 집계 전달은 _signal 접미(tb_news_signal), 뷰는 v_ 접두.
  7. 신호(_signal) 테이블은 append 불변. update 금지 — 최신 상태는 (ticker, cycle_ts DESC) 뷰/인덱스로 읽는다.
  8. PK 규칙 — 시각 컬럼·ticker 단독을 PK로 쓰지 않는다. 사이클 append 테이블은 대리키 PK(tns_id·tds_id) + UNIQUE(ticker, cycle_ts)(멱등), 일봉 테이블은 (날짜, ticker) 자연 복합 PK.

🧬 계보(provenance) 공통 컬럼이란먼저 읽기 — 아래 표들의 "(계보)" 행이 이것설명

계보 = 데이터의 "출생 기록"이다. 마트 소고기에 산지·도축일·가공공장 라벨이 붙듯, LLM이 만든 점수마다 원재료는 어디서 왔고(source·source_ref·captured_at) · 어떤 증거에서 파생됐고(evidence_id·parent_evidence_ids) · 누가 어떤 레시피로 만들었나(model_provider·model_name·prompt_version·policy_version·input_hash)를 함께 기록한다. cycle_ts가 "시간표의 칸 번호"라면 계보는 "출생·제조 기록"으로, 서로 다른 것이다.

쓰임새 3가지 — ① 점수가 이상하면 원문 링크·당시 프롬프트 버전으로 역추적 ② 회고 때 "판단이 틀린 게 전략 탓인가, 옛 프롬프트 탓인가" 구분 ③ model_provider=mock이면 테스트 가짜 점수임이 데이터에 박혀 있어 통계 오염 방지. 아래 각 표에서 (계보) 표시가 붙은 행이 이 10컬럼이다.

라이프사이클 — 별도 테이블이 아니라 각 행에 내장 (LLM 산출 테이블 — 공시·뉴스 원본·집계 + 회고 스냅샷, M2부터 전략·반박도 포함): 데이터와 같은 INSERT로 태어나(원자적 — 계보 없는 데이터 불가) → append 불변으로 얼어붙고 → 평상시엔 아무도 안 읽다가 역추적·회고·감사 때만 조회 → 행과 함께 영구 보존. 행 간 족보는 parent_evidence_ids가 그래프로 연결한다.

03 일일 스크리너오늘의 50 선발as-built
지표 기반 5버킷 스코어링 → 버킷별 상위 + 종합 랭크로 50종목. 픽 게이트 score ≥ 0.70. M2: LLM 심층 대상 = 상위 20 ∪ 보유 종목.
06 대표 기사 선발뉴스 집계의 얼굴 1건as-built → M2 개정
as-built: 관련성 점수(제목50·스니펫25·회사명 매칭) 최상위, 동률 시 published_at 최신 → canonical_url → index.
M2 개정(하이브리드): ① 관련성 필터로 오귀속 차단(통과/탈락) → ② 통과분 중 importance × 신뢰무게(w) 최상위 1건, 동률 published_at 최신 → id. 신뢰무게 w = 확신도 × 확인계수(확정 1.0 / 소문 0.5) × 매체등급.
06 importance vs peak_importance대세 vs 한 방M2
importance = 신뢰가중 평균(전체 흐름의 대세). peak_importance = 신뢰 검증분 최댓값(큰 단일 촉매 경보). 예: 로이터 0.90 + 블로그 4건 0.20 → importance 0.38, peak 0.90. 판단: 둘 다 높으면 강한 후보, peak만 높으면 재확인 후 보류, importance만 높으면 우선순위 하향, 둘 다 낮으면 제외.
07 전략 — conviction·게이트매수/매도 판정as-built → M2 게이트 추가
as-built: conviction = round((technical + disclosure + news + LLM점수) / 4, 3) · 매수 = is_daily_pick AND conviction ≥ 문턱 AND not blockers (문턱 as-built 0.65~0.70).
M2 게이트 3종 추가: ① source_trust < 0.55인 뉴스 신호는 투표 박탈 ② 매크로 risk_score→감점표(최대 −0.40) ③ 강악재 sentiment ≤ 0.15 차단. 매수 문턱 성향별(공격 0.65 / 안전 0.75).
M2 매도(R7-1): 보유 종목은 진입가·보유일·손익 맥락과 함께 buy/hold/sell 판정. 청산 3층 = 브래킷 · 시간(10영업일+논지 미실현) · 논지 붕괴(하드 이벤트 코드 직행 / 점수 악재 07→08).
08 반박(Critic) — 판정 순서적대적 검증as-built → M2 배선 수정
판정 층 순서: quality_gate(데이터·스키마 불량) → hard_rule(가격 이상치 등) → llm(논리 반박) → gate. 승인 confidence ≥ 0.70, 미만 reject. 과신 conviction ≥ 0.90 → 문턱 0.80.
M2: 현재 08이 시세를 합성(close_prev=현재가)해 30% 가드가 무력 → 07 박제 시세 실전달 + 허용오차 2% 검증으로 수정. 매도 판단도 검증(패닉 매도 차단).
09 리스크·사이징·서킷계좌별 실행as-built → M2 서킷
as-built 브래킷: stop = 진입가 × (1 − 0.15) · take = 진입가 × (1 + 0.20) (STOP_FRACTION 0.15 / TAKE 0.20). 일일 주문 한도 예약 구조 있음.
M2 서킷브레이커 4종(성향별): 동시 보유(10/5) · 종목당 비중(20%/10%) · 일일 손실 한도(−4%/−2%) 도달 시 당일 신규 매수 전면 중단(익영업일 해제) · 최소 현금(10%/30%). + late_entry(ret_5d ≥ 15%/12% 금지) · 주문 직전 halted 체크.

안전 기본값: config mvp2.schedule.enabled=false — W0 수동 운용 중 자동 발화 방지, 운영 전환 시 yaml 한 줄. 캘린더는 exchange-calendars(XNYS) — 서머타임 전환일(3/8·11/1) 검증 포함.

공시·뉴스·전략 신호는 전부 (trade_date, ticker)로 오늘의 픽(tb_daily_pick)에 FK 연결 — 여기서 trade_date는 마지막 거래 세션의 봉 날짜(데이터 기준)다. 주말·프리마켓엔 시계 날짜(cycle_ts.date())가 이미 다음 날로 넘어가 있어 시계 기준으로 쓰면 FK가 깨진다(토요일 드라이런에서 실증).
