## 프로그램 한눈에 보기

Quantinue는 **실제 시장 데이터에서 투자 후보를 수집·분석하고, 공격형·보수형 전략가와 독립 비평가(critic), 코드 안전장치를 거쳐 MockBroker 모의 계좌에서 주문·청산하며, 전 과정을 원장·웹·알림으로 추적하는 AI 보조 모의투자 운영 시스템**이다.

실제 증권 계좌에 주문하지 않으며 수익을 보장하지 않는다. 이 보고서의 성과는 투자수익이 아니라 기능과 운영 안전성에 대한 검증 결과다.

`입력(시장 데이터) → 처리(후보 수집·분석) → 판단(전략가·비평가·코드 안전장치) → 모의 집행(MockBroker 주문·청산) → 관측(원장·웹·알림)`

- **관리자 관점:** 작업 실행 상태, 계좌, AI 비용, 위험 방어, 오류와 알림을 확인하고 계좌 운영 상태를 관리한다.
- **사용자 관점:** 자기 모의 계좌의 자산, 보유 종목, 체결 이유와 시장 기준 대비 결과를 읽기 전용으로 확인한다.

### 이 보고서에서 확인할 것

1. Quantinue가 어떤 데이터를 받아 어떻게 판단하고 모의 집행하는가
2. 1주차의 시연형 파이프라인이 2주차에 무엇으로 확장됐는가
3. 무엇이 완료됐고, 무엇이 운영 확인 또는 다음 개발로 남았는가

### 용어 안내

| 용어 | 쉬운 뜻 |
|---|---|
| JOB(잡) | 정해진 시간에 독립적으로 실행되는 하나의 작업 단위 |
| critic | 전략가의 제안을 그대로 따르지 않고 근거와 위험을 다시 확인하는 독립 비평가 |
| 원장 | 작업·판단·비용·주문 결과를 나중에 추적할 수 있게 쌓는 기록 |
| fail-closed | 정보가 없거나 오류가 나면 안전을 위해 신규 주문을 막는 방식 |
| dead-man 감시 | 앱이 아무 알림도 보내지 못할 정도로 멈췄을 때 외부에서 침묵을 감지하는 장치 |
| lease·fencing | 한 작업의 실행 권한과 세대를 확인해 오래된 실행이 중복 처리하지 못하게 하는 장치 |
| Gate(운영 관문) | 코드를 실제 운영에 켜기 전에 통과해야 하는 관측·검증 조건 |

### 완전 자동화의 의미와 현재 수준

이 보고서에서 완전 자동화란 하루 한 번 분석 작업을 실행하는 데 그치지 않고, **거래일 시작부터 데이터 수집·판단·모의 주문·관측·장애 복구까지 사람의 반복 조작 없이 이어지는 것**을 뜻한다. 실제 증권 계좌를 운용하는 자동매매가 아니라 **MockBroker 모의 계좌의 완전 자동 운영**이 목표다.

완전 자동화 범위에는 거래일과 정해진 시각에 맞춘 자동 기동, 데이터 수집과 후보 선별, 전략가와 critic 판단, 코드 기반 위험·예산·거래 가능 여부 검사, MockBroker 자동 매수·청산·배분이 포함된다. 결과는 원장·웹·Telegram·외부 생존 신호로 관측하며, 장애 뒤에도 같은 작업을 중복하지 않고 이어서 실행해야 한다.

| 현재 수준 | 포함 범위 | 사람의 개입 |
|---|---|---|
| **현재 자동화됨** | 거래일·슬롯별 일일 JOB 기동, 데이터 수집·후보 선별, 전략가+critic 판단, 위험·예산·거래 가능 여부 검사, MockBroker 매수·청산·배분, 원장·웹·Telegram·외부 heartbeat 구현, 장애 후 멱등 재개 | 계좌와 정책 설정, 비정상 경보 대응·외부 신호 확인 |
| **구현됐으나 Gate 후 활성화** | 1분 가격 감시 기반 방어, 가격 급변·정기 LLM 재판단, 보유 종목 웹소켓 | 실제 운영 원장 확인 후 단계별 활성화 승인 |
| **아직 개발 중** | 공시 30분·뉴스 15분·와이어 10분 증분 수집, 새 관련 사건만 LLM 재판단·모의 집행 | 사건 경로 완료·검증 승인, AWS 이전 여부 결정 |

최종 목표 흐름은 다음과 같다.

`일일 전체 체인 1회 + 가격 1분 + 공시 30분·뉴스 15분·와이어 10분 증분 수집 → 새 관련 사건이 있을 때만 LLM 판단 → 자동 모의 집행 → 실패 알림과 안전한 재개`

보고 기준일 현재 Todo 13 이후의 장중 사건 경로는 미완료다. 보고서 작성 이후의 후속 진행은 별도 개발 현황에서 관리하며, 이 기준일의 자동화 수준과 섞지 않는다.

## 1. 주간 요약

2주차의 결론은 1주차 시연형 흐름을 **독립 실행과 재시도가 가능한 14종 JOB 기반 운영 구조**로 확장했다는 것이다.

NYSE 거래일과 뉴욕 슬롯을 기준으로 작업을 등록하고, PostgreSQL 원장을 통해 같은 작업의 중복 실행을 막도록 구성했다. 로컬 운영 DB는 5445 포트로 통일했다.

데이터 수집부터 공격형·보수형 전략가와 critic의 판단, 모의 주문·청산·배분까지 일일 체인을 연결했다. 관리자 화면과 사용자 화면은 역할에 따라 분리했다.

주문은 외부 증권 계좌로 전송하지 않으며, **MockBroker를 이용한 모의 체결**만 수행한다.

장중 대응은 가격 감시 M1~M7을 코드로 구현했다. 정규장 1분 가격 확인, 손절·익절의 같은 확인 주기 내 모의 청산, 가격 급변·정기 재판단, 장중 배분, 관제 화면, 웹소켓과 폴링의 혼합 구조를 포함한다.

다만 운영 활성화는 단계적으로 진행한다. 가격 폴링은 실제 적용 확인이 남았고, LLM 재판단과 웹소켓은 비활성 상태다.

운영 안정성을 위해 8020을 일일 잡·감시·생존 신호의 단일 소유자로 두고, 8021은 화면만 제공하는 개발 인스턴스로 분리했다. 예산 잠금, 중복 실행 방지, 부분 실패 재개, 주문 직전 fail-closed, Telegram과 외부 생존 감시도 반영했다.

Wi-Fi 변경과 서버 이동으로 실제 운영 관측이 일부 누락됐다. 따라서 코드 완료와 무인 운영 검증을 같은 의미로 보지 않는다.

남은 핵심은 장중 새 공시·뉴스·와이어를 증분 수집해 관련 종목만 재판단하는 사건 경로다. 계획은 확정됐지만 보고 기준일 현재 완료 항목은 아니다.

### 1.1 1주차 대비 핵심 변화

| 비교축 | 1주차 | 2주차 현재 상태 |
|---|---|---|
| 실행 구조 | 01~11단계 단일 파이프라인 | 14종 JOB으로 분리하고 잡별 재시도·실패 격리 구현 |
| 스케줄 | 수동 또는 단일 실행 중심 | NYSE 거래일·뉴욕 슬롯과 DB 원장 기준의 멱등 실행 구현 |
| 투자 성향 | 공격형 1종 우선 검증 | 공격형·보수형 전략가와 성향별 critic 판단 구현 |
| 데이터 범위 | SEC 공시와 뉴스 중심 | 유니버스·일봉·SPY·SEC 공시·Form 4·뉴스·와이어·FRED로 확대 |
| 주문·계좌 | 외부 전송 없는 로컬 모의 주문 | MockBroker 브래킷 매수·청산·배분과 주문 직전 거래 가능 여부 fail-closed 구현·독립 검증 |
| 운영 화면 | 단일 시연·단계 추적 화면 | 관리자·사용자 화면 분리, 비용·작동 로그·SPY 비교와 권한 경계 구현 |
| 장중 대응 | 별도 장중 감시 없음 | 가격 watch M1~M7 코드 구현. 가격 폴링은 적용 확인 중이고 LLM 재판단·웹소켓은 비활성 |
| 알림·생존 감시 | 운영자가 화면과 실행 결과를 직접 확인 | Telegram 내부 알림과 외부 dead-man 감시 구현. 실제 연속 수신은 추가 관측 필요 |
| LLM·비용 | 로컬 LLM 중심의 흐름 검증 | OpenAI 실측과 사용량 원장, Standard 요율 보정, 일일 `$3` 하드캡 적용 |
| 배포·실행 경계 | 로컬 MVP 실행 | 8020 단일 작업 owner와 8021 web-only 개발 화면 분리. AWS는 최소안만 계획 |
| 남은 핵심 | 정기 자동화와 데이터 품질 보완 | 장중 신규 공시·뉴스·와이어 사건 경로가 Todo 13 이후 개발 단계이며 아직 미완료 |

비교표의 “구현”은 코드와 독립 검증이 끝난 범위를 뜻하며, “적용 확인 중”은 실제 운영 원장 관측이 더 필요한 범위를 뜻한다. 장중 사건 경로와 AWS 이전은 완료가 아니라 각각 개발 단계와 계획 단계다.

## 3. 주요 진행 내용

구현의 핵심은 수집·판단·모의 집행을 독립 작업으로 나누고, 각 결과를 추적 가능한 기록으로 연결한 것이다.

### 3.1 잡 기반 실행 구조와 데이터 원장

- 기존 단일 러너를 유니버스, 시세, 공시, 뉴스, 거시, 선별, 판단, 청산, 배분 등 14종 잡으로 분리했다.
- NYSE 거래일과 뉴욕 날짜의 계획 슬롯을 기준으로 실행하며, `tb_job_run` 원장이 같은 슬롯의 중복 실행을 방지한다.
- 등록된 잡 수와 특정 날짜의 성공 행 수는 다를 수 있다. 조건부 잡이나 실행 시점이 아닌 잡은 그날 행이 생기지 않기 때문이다.
- DB 연결 기준을 5445로 통일하고, 스키마와 migration의 반복 적용 및 복구 절차를 정리했다.
- 재시도, 굳은 잠금 해제, 실패 격리를 도입해 한 작업의 실패가 전체 운영 이력을 덮지 않도록 했다.

### 3.2 수집·판단·비평 체인 확장

- NASDAQ 유니버스와 보유 종목 이월, Alpaca 일봉·최근가·뉴스, SEC 공시·Form 4, FRED 금리, RSS/Atom 와이어를 수집 경로에 연결했다.
- 저장된 데이터를 바탕으로 기술 랭킹과 당일 후보를 만들고, 공격형·보수형 전략가가 각각 제안하도록 확장했다.
- 전략가의 제안은 critic의 독립 반박과 승인·기각을 거치며, 종목별 실패는 다른 종목 판단과 분리한다.
- 판단 근거, 모델, 토큰, 비용, 입력 계보를 원장에 남겨 사후 추적이 가능하도록 했다.

### 3.3 모의 주문·청산·배분

- 승인된 후보에 현금, 포지션, 일일 신규 매수 한도와 위험 정책을 적용해 배분한다.
- 매수는 손절·익절 조건을 함께 가진 브래킷 형태로 원장에 기록한다.
- 손절, 익절, 투자 명제 붕괴, LLM 약세 판단, 보유기간 만료를 모의 청산 경로에 연결했다.
- 새 매수 주문 제출 직전에 거래 가능 여부를 다시 확인하고, 거짓·오류·시간 초과·불명 상태에서는 주문을 만들지 않는 fail-closed 방어를 반영했다.
- 모든 체결은 **MockBroker의 모의 체결**이며 실거래 주문이나 실제 투자 집행이 아니다.

### 3.4 관리자·사용자 웹

관리자 화면은 잡 체인, 계좌 총람, LLM 지출, 방어선 발동, 장중 판단, 작동 로그와 계좌 관리를 제공한다. 사용자 화면은 자기 계좌의 총자산·현금·보유·수익률, 체결과 연결된 전략가 근거와 critic 반박, 청산 이유, SPY 대비 성과를 읽기 전용으로 보여준다.

관리자와 사용자 구역을 인증·권한으로 분리하고, 사용자 역할의 쓰기 엔드포인트가 없음을 테스트로 고정했다. 화면의 성과 표시는 투자수익을 입증하는 결과가 아니라 모의 계좌 원장과 벤치마크 표시 기능의 동작 확인 자료다.

### 3.5 장중 가격 감시 M1~M7

| 단계 | 구현 내용 | 보고 기준 상태 |
|---|---|---|
| M1 | 정규장·휴장 세션 게이트와 no-op 감시 tick | 구현·사용 후보 |
| M2 | 최근가 배치 조회와 mock fixture | 구현·사용 후보 |
| M3 | 브래킷 평가 이관과 같은 tick 모의 청산 | 구현 완료 |
| M4 | ±5% 가격 트리거, 30분 cooldown, 전략가+critic 재사용, 장중 매도 | 구현 완료·비활성 |
| M5 | 뉴욕 10:00·12:45·15:15 정기 스윕과 장중 매수 | 구현 완료·비활성 |
| M6 | 관리자 장중 판단 카드 | 구현 완료 |
| M7 | 보유 종목 웹소켓과 나머지 1분 폴링 혼합 | 구현 완료·비활성 |

손절·익절 방어는 LLM과 분리된 결정론적 1분 루프가 담당한다. 비용이 드는 LLM 재판단은 가격 사건이나 정해진 스윕에서만 수행하며, 30분 cooldown과 예산 가드를 공유한다. 웹소켓은 보유 종목에 한해 마지막 단계에서 활성화하고, 장애 시에도 1분 폴링 방어를 유지하는 구조다.

### 3.6 운영 안정성·비용·관측

- 8020은 잡·WatchRunner·heartbeat를 소유하는 단일 운영 인스턴스, 8021은 백그라운드 작업이 없는 web-only 개발 인스턴스로 분리했다.
- sweep lease 갱신, attempt fencing, 재시작 멱등성, 부분 실패 재개로 중복 유료 판단과 중복 주문을 방지했다.
- 모든 유료 판단은 사용량 원장을 거치며, 동시 예약을 포함해 일일 `$3` 하드캡에서 fail-closed한다. 매도성 판단을 위해 마지막 20% 예산을 예약한다.
- 2026-07-22 OpenAI 실측은 **43콜, 입력 70,491 토큰, 출력 8,745 토큰**이었다.
- 당시 원장의 약 `$0.046110`은 Standard가 아닌 낮은 요율을 사용한 값이므로, Standard 기준 **약 `$0.092220`**으로 보정했다.
- Telegram은 잡 실패·기동·일일 요약·방어선 등을 알리고, 외부 생존 감시는 앱 자체의 침묵을 감지하도록 역할을 나눴다.
- Wi-Fi와 서버 위치 변경으로 일부 heartbeat 및 무인 슬롯 관측이 누락됐다. 따라서 설정·코드 존재만으로 운영 정상 판정을 내리지 않고, 8020 owner·최근 tick·DB 원장·외부 수신 기록을 함께 확인한다.

## 7. 리스크 및 유의사항

가장 중요한 유의사항은 코드 구현 완료가 실제 운영 검증 완료나 투자수익 입증을 뜻하지 않는다는 점이다.

- 현재 주문과 체결은 모두 **MockBroker 모의 처리**다. 실거래 성과나 투자수익을 증명하지 않는다.
- M1~M7의 코드 완료와 운영 활성화는 다르다. 가격 폴링, LLM 재판단, 웹소켓은 정해진 Gate를 통과하며 순차적으로 켜야 한다.
- 장중 신규 공시·뉴스·와이어 사건 경로는 아직 미완료다. 현재 공시·뉴스 관련 일일 잡은 거래 슬롯마다 1회 실행된다.
- 최종 권고 주기는 **일일 전체 체인 1회, 공시 30분, 뉴스 15분, 와이어 10분, 가격 1분**이다. 새 관련 사건이 0건이면 사건 기반 LLM 호출도 0회여야 한다.
- 기사 원문은 수정하지 않는 버전으로 저장하고, 출처·중요도·종목 관련성은 결정론 필터가 먼저 판단한다. 정규화 본문 12,000자 이하는 직접 증거팩으로 전달하고, 초과 문서만 해시·모델·프롬프트 버전 기준으로 요약을 캐시한다.
- 기사 내용은 신뢰할 수 없는 입력으로 취급하며, 문서 안의 명령·링크·도구 호출 문구를 시스템 지시로 실행하지 않는다.
- 예상 LLM 비용은 보통 **하루 `$0.40~0.50`, 10거래일 `$4~5`**이고 코드상 하드캡은 **하루 `$3`**다. 이는 사건 빈도와 호출량에 따라 달라지는 추정값이다.
- AWS는 아직 계획 단계다. Lightsail 2GB 한 대의 **월 `$13~15`**는 최소 구성 추정값이며, 실제 자원 생성이나 이전 완료를 의미하지 않는다.
- 비밀값, 계정, 이메일, 실제 heartbeat 주소와 운영 IP는 보고서와 일반 로그에 남기지 않는다.

## 8. 차주 계획

차주는 전체 분석을 반복하기보다 새 사건만 가볍게 수집하고 관련 종목만 다시 판단하는 경로를 완성하는 데 집중한다.

1. **불변 사건 저장 계약 구현**<br>
   공시·기사 원문 버전, cursor, event, 처리 영수증을 정의하고 재기동·중단에도 덮어쓰기나 중복 처리가 없도록 한다.

2. **장중 증분 수집과 결정론 필터 연결**<br>
   SEC 공시는 30분, 일반 뉴스는 15분, 와이어는 10분마다 증분 수집하고, 새 사건·관련 종목·중요도·중복 여부를 코드로 먼저 판정한다.

3. **증거팩과 선택적 요약 구현**<br>
   12,000자 이하 원문은 직접 근거를 포함한 제한된 증거팩으로 전달하고, 더 긴 원문만 캐시 요약과 정확한 원문 구간을 함께 사용한다.

4. **표적 전략가·critic·모의 집행 연결**<br>
   관련된 보유 종목과 당일 후보만 기존 전략가·critic에 한 번 연결하고, 판단이 바뀐 경우에만 기존 모의 청산·배분 경로를 멱등하게 호출한다.

5. **운영 Gate와 관측 보강**<br>
   무사건 0콜, 중복 사건 0콜, 관련 사건 1회 판단을 8021 mock과 임시 DB에서 검증한 뒤, 8020에서 가격 폴링·일일 OpenAI 슬롯·사건 경로를 순서대로 관측한다.

6. **AWS 이전 여부 재판정**<br>
   로컬 정규장 검증과 복구 절차가 통과한 뒤에만 Lightsail 최소안의 이전·롤백 체크리스트를 확정한다. 다음 주 계획에 AWS 자원 생성을 포함하지 않는다.

## 9. 주간 결론

2주차에는 Quantinue를 단일 시연 파이프라인에서 **독립 잡, 원장, 두 성향의 전략가와 critic, MockBroker 모의 매매, 관리자·사용자 웹, 장중 가격 방어를 갖춘 운영형 MVP-2**로 확장했다. 동시에 단일 owner, 비용 상한, lease·fencing, 거래 가능 여부 재확인, 내부 알림과 외부 생존 감시를 통해 자동 실행이 잘못되거나 침묵할 때의 방어선을 강화했다.

OpenAI 실측과 요율 보정을 통해 비용을 추정이 아닌 원장 기준으로 관리하기 시작했고, 장중 가격 감시 M1~M7의 코드 기반도 마련했다. 그러나 Wi-Fi와 서버 이동으로 연속 실장 관측이 충분하지 않았으므로, “구현됨”과 “운영에서 검증됨”을 구분한다.

현재 상태는 **“일일 전체 체인과 가격 기반 장중 대응의 핵심 구현은 완료됐고, 운영 Gate 검증과 장중 신규 사건 경로는 진행 중”**이다. 다음 주의 핵심은 전체 체인을 자주 반복하는 것이 아니라 새 공시·뉴스·와이어만 증분 수집하고, 관련 사건이 있을 때에만 제한된 근거로 전략가와 critic을 호출하는 것이다. 이 경로가 무사건 0콜, 중복 주문 0건, 비용 상한 준수를 실제 원장으로 입증해야 MVP-2의 이번 범위를 완료로 판정할 수 있다.
