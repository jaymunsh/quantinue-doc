# 웹 2면 계획 — 관리자 ERP · 유저 포털 (2026-07-20)

> 2차 확정 결정 "**전체 자동화 + 관리자/유저 2면 서비스**"(phase1-decisions §1)의
> 두 번째 절반. 첫 번째 절반(자동화)은 JOB 12종으로 완료됐고, 이 문서는
> 구 M9(관리자 ERP)·M10(유저 포털) 기획을 **잡 세계로 번역**해 승계한다.
> M5~M11 번호 체계는 폐기됐지만 이 기획은 폐기된 적이 없다.

## 승계하는 확정 사항 (phase1-decisions — 되묻지 말 것)

- **1유저 = 1계좌** · **셀프 가입 없음** — 유저 계정 생성은 관리자만
- **유저 화면 = 조회 전용** — 유저의 쓰기 엔드포인트 **0**(성향 지정도 관리자)
- 근거 서사 유지: "LLM 출력이 검증 없이 돈에 닿는 경로 0개" — 유저 read-only는
  이 서사의 일부이자 구현량 통제
- 계좌 구성: 데모 5(공격 $150K·$100K·$5K / 안전 $100K·$5K) + 테스트 2 —
  `scripts/provision_accounts.py`가 이미 시드하며 tb_user 연결도 그 스크립트 소관

## 이 계획에서 확정하는 결정

**W-D1. 인증은 2단계로 나눈다.** 1단계 = 로그인(비밀번호 해시) + 서명 세션
쿠키 + role 가드. TOTP(`pyotp`)는 2단계로 미룬다 — 로컬 데모에서 OTP는
마찰만 더하고, 원안(M9-1)의 목표는 "역할 분리와 소유권 검증"이지 2FA 자체가
아니다. 원안 폐기가 아니라 순서 조정: 페이퍼 전환(R1) 전 게이트에 TOTP를
포함시킨다.

**W-D2. 관제실은 관리자 구역으로 들어간다.** 지금 `GET /`는 계좌 전체의
현금·보유·판단을 보여준다 — 유저별 소유권 검증과 양립하지 않는다. 단
**부트스트랩 예외**: `tb_user`에 행이 0이면 로그인 없이 열린다(첫 설치·개발
편의). 관리자 계정이 생기는 순간 잠긴다.

**W-D3. 수동 컨트롤은 "슬롯 재실행" 하나로 준다.** 구 R9의 버튼들(사이클
실행·특정 역할 실행·체크포인트 재개·catch-up)은 구 러너의 스테이지 재개
전제 위에 있었다. 잡 세계에선 멱등이 재개를 대체한다 — 실패한 잡의
`tb_job_run` 슬롯 행을 지우고 다음 틱이 오게 하면 그게 재실행이다.
버튼 하나(= 슬롯 삭제 + 즉시 틱 트리거), 완료된 슬롯이면 no-op 경고.

**W-D4. SPY 대비는 벤치마크 수집 잡이 선행이다.** `tb_benchmark_price`는
생산자가 없다. W2의 "내 계좌 홈"은 1차로 **계좌 곡선(tb_account_equity_daily)**
만으로 출시하고, SPY 비교는 벤치마크 수집 잡(일 1회, daily_bars와 같은
어댑터로 SPY 한 종목)을 붙인 뒤 얹는다. 없는 데이터를 화면이 지어내지 않는다.

**W-D5. 매니저 리포트·투명성 리포트(M10-3/4)는 후순위.** 원료가 T+5 리뷰의
lesson인데 lesson 생성(LLM) 배선이 아직 없다. 1차 유저 포털 범위는
홈 + 거래 타임라인 + 리스크 상태 3장이다.

### 착수 실사에서 드러난 것 (2026-07-20)

**① `tb_user`에 비밀번호 컬럼이 없다.** → W1-0 신설.

**② 1유저=1계좌는 이미 DB가 강제하고 있다.** `schema.sql:241`:
```sql
CREATE UNIQUE INDEX tb_account_user_id_key ON tb_account(user_id) WHERE user_id IS NOT NULL;
```
새로 만들 게 없고, **W1-5 시드가 이 제약의 첫 실전 소비자**다. 지금 시드되는
계좌 7개는 전부 `user_id`가 비어 있다(`db/provisioning.py:45-51`의 `AccountWrite`가
담지 않는다).

**③ 쓰기 엔드포인트가 이미 하나 무방비다.** 리뷰 POST(`api/reviews.py:38-39`)는
`access is not None`일 때만 토큰을 검사하는데 `access`는 `trading_enabled`일 때만
만들어진다(`main.py:93-97`). **거래가 꺼진 구성에서는 인증 없이 열린다.**
W1-4에서 "admin 세션 **또는** 유효 토큰"으로 닫는다 — 토큰 경로를 남기는 이유는
스케줄러·스크립트가 이 엔드포인트를 부를 수 있기 때문이다.

**④ 웹 계층에 `Depends` 패턴이 없다.** 스토어는 `create_app`이 만들어 핸들러가
클로저로 잡는다(`main.py:138,149`). 인증도 같은 관용구를 따른다 — 새 DI 체계를
들이지 않는다.

### W1 확정 결정 (문성혁 확정, 2026-07-20)

| | 결정 | 이유 |
|---|---|---|
| 해시 | `argon2-cffi` | stdlib `hashlib.scrypt`는 의존성이 0이지만 KDF 파라미터를 코드 리터럴로 박아야 한다 — 문턱을 코드가 소유하게 된다 |
| 세션 | Starlette `SessionMiddleware` | ⚠️ **서명만 하고 암호화하지 않는다** — `user_id`·`role`이 base64로 보인다(변조는 불가). 로컬 데모에서 서버측 세션 테이블을 새로 만드는 값보다 싸다 |
| 세션 키 | `session_secret: SecretStr \| None = None` | None이면 기동 시 **랜덤 생성 + 경고 로그**. 필수 필드로 두면 기존 523 테스트가 전부 깨지고, 조용히 고정 개발키를 박으면 그게 프로덕션에 그대로 간다 |
| 부트스트랩 | `tb_user` 0행 → **미인증을 admin으로 취급** | memory 모드(유닛 테스트)는 `tb_user`를 조회할 수 없어 자연히 0행이 된다 → **기존 523 테스트가 손대지 않아도 통과한다** |
| 시드 | admin 1 + `user1`~`user5` → DEMO 5 | `TEST-*` 2개는 소유자 없이 둔다(관리자 테스트용) |
| 관제실 경로 | W1에서는 **가드만**, 이동은 W3-1 | 경로 이동과 인증을 한 커밋에 섞지 않는다 |

**⚠️ dev DB(5445)는 W1-5를 돌리는 순간 관제실이 잠긴다.** 예고된 거동이다.
세션 쿠키라 한 번 로그인하면 유지되므로 개발 마찰은 거의 없다. 시드는 W1-4까지
끝낸 뒤 마지막에 돌린다.

## 순서와 규모

W1 → W2 → W3 → W4 순. W2와 W3은 W1 위에서 독립적이라 병렬 가능하지만,
소유권 검증(W2)이 가드(W1-4)의 첫 실전 검증이므로 W2 먼저를 권장.

착수 규칙(playbook M9 원문): **첫 태스크 = 와이어프레임**. 아래 초안을
착수 세션에서 확정하고 구현에 들어간다.

## 와이어프레임

> W1 대상(공통 셸·`/login`·`/me` 뼈대)은 **확정**이고, W2·W3 화면은 초안이다.
> 텍스트 와이어프레임으로는 감이 안 오는 것이 정상이라, W1-4까지 되면 앱을
> 8020에 띄워 **실제 화면을 보고 다듬는다**. 스타일은 새로 만들지 않고
> `web/static/dashboard.css`의 디자인 토큰(`:root` 커스텀 속성)을 그대로 쓴다.

### 공통 셸 (`base.html` — 신설) ✅ 확정
```
┌ QUANTINUE ─────────── [모드칩] ─── 정창욱 · 로그아웃 ─┐
```
지금은 topbar가 `pipeline.html` 한 장에 박혀 있다. W2·W3에서 화면이 3장 더
붙는데 복사하면 **한쪽만 고쳐지는 날이 온다** — `pipeline.html`이 이걸
상속하게 바꾼다.

### 로그인 (`/login`) ✅ 확정
```
┌──────────────────────────────┐
│  QUANTINUE                   │
│  아이디   [            ]     │
│  비밀번호 [            ]     │
│  [ 로그인 ]                  │
│  ⚠ 아이디 또는 비밀번호가     │  ← 실패 시. 계정 없음·비활성·
│    올바르지 않습니다.         │     오답 전부 같은 문장
│  계정은 관리자가 발급합니다.  │  ← 가입 링크 없음을 화면이 말한다
└──────────────────────────────┘
```

### 유저 — `/me` 뼈대 (W1 범위) ✅ 확정
```
┌ QUANTINUE ───────────── 정창욱 · 로그아웃 ─┐
│ 내 계좌  DEMO-AGGRESSIVE-01 · 공격형        │
│ (자산·곡선·보유는 W2-1에서)                 │
└─────────────────────────────────────────────┘
```
W1에서 이걸 만드는 이유: 유저를 시드해놓고 로그인 후 갈 곳이 없으면
**가드를 실전으로 검증할 수 없다.** 그리고 이 한 줄이 이미 소유권
쿼리(세션 `user_id` → 계좌)의 첫 소비자라 유령이 아니라 W2-1의 바닥이다.

### 유저 — 내 계좌 홈 (`/me`)
