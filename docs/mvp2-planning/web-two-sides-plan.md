# 웹 2면 계획 — 관리자 ERP · 유저 포털 (2026-07-20)

> 2차 확정 결정 "**전체 자동화 + 관리자/유저 2면 서비스**"(phase1-decisions §1)의
> 두 번째 절반. 첫 번째 절반(자동화)은 JOB 12종으로 완료됐고, 이 문서는
> 구 M9(관리자 ERP)·M10(유저 포털) 기획을 **잡 세계로 번역**해 승계한다.
> M5~M11 번호 체계는 폐기됐지만 이 기획은 폐기된 적이 없다.

## 승계하는 확정 사항 (phase1-decisions — 되묻지 말 것)

- **1유저 = 1계좌** · **셀프 가입 없음** — 유저 계정 생성은 관리자만
- **유저 화면 = 조회 전용** — 유저의 쓰기 엔드포인트 **0**(성향 지정도 관리자)
- 근거 서사 유지: "LLM 출력이 검증 없이 돈에 닿는 경로 0개" — 유저 read-only는
  이 서사의 일부이자 구현량 통제
- 계좌 구성: 데모 5(공격 $150K·$100K·$5K / 안전 $100K·$5K) + 테스트 2 —
  `scripts/provision_accounts.py`가 이미 시드하며 tb_user 연결도 그 스크립트 소관

## 이 계획에서 확정하는 결정

**W-D1. 인증은 2단계로 나눈다.** 1단계 = 로그인(비밀번호 해시) + 서명 세션
쿠키 + role 가드. TOTP(`pyotp`)는 2단계로 미룬다 — 로컬 데모에서 OTP는
마찰만 더하고, 원안(M9-1)의 목표는 "역할 분리와 소유권 검증"이지 2FA 자체가
아니다. 원안 폐기가 아니라 순서 조정: 페이퍼 전환(R1) 전 게이트에 TOTP를
포함시킨다.

**W-D2. 관제실은 관리자 구역으로 들어간다.** 지금 `GET /`는 계좌 전체의
현금·보유·판단을 보여준다 — 유저별 소유권 검증과 양립하지 않는다. 단
**부트스트랩 예외**: `tb_user`에 행이 0이면 로그인 없이 열린다(첫 설치·개발
편의). 관리자 계정이 생기는 순간 잠긴다.

**W-D3. 수동 컨트롤은 "슬롯 재실행" 하나로 준다.** 구 R9의 버튼들(사이클
실행·특정 역할 실행·체크포인트 재개·catch-up)은 구 러너의 스테이지 재개
전제 위에 있었다. 잡 세계에선 멱등이 재개를 대체한다 — 실패한 잡의
`tb_job_run` 슬롯 행을 지우고 다음 틱이 오게 하면 그게 재실행이다.
버튼 하나(= 슬롯 삭제 + 즉시 틱 트리거), 완료된 슬롯이면 no-op 경고.

**W-D4. SPY 대비는 벤치마크 수집 잡이 선행이다.** `tb_benchmark_price`는
생산자가 없다. W2의 "내 계좌 홈"은 1차로 **계좌 곡선(tb_account_equity_daily)**
만으로 출시하고, SPY 비교는 벤치마크 수집 잡(일 1회, daily_bars와 같은
어댑터로 SPY 한 종목)을 붙인 뒤 얹는다. 없는 데이터를 화면이 지어내지 않는다.

**W-D5. 매니저 리포트·투명성 리포트(M10-3/4)는 후순위.** 원료가 T+5 리뷰의
lesson인데 lesson 생성(LLM) 배선이 아직 없다. 1차 유저 포털 범위는
홈 + 거래 타임라인 + 리스크 상태 3장이다.

## Phase W1 — 인증과 뼈대

| # | 태스크 | 상세 |
|---|---|---|
| W1-1 | 의존성·해시 | 비밀번호 해시(argon2 계열) + 세션 서명. 선택지는 착수 시 확정 — 잠정 `argon2-cffi` + `itsdangerous` 방식 서명 쿠키 |
| W1-2 | tb_user 배선 | 스토어에 유저 읽기·검증 경로. **첫 소비자 탄생** — open-items §2의 `tb_user` 항목이 여기서 닫힌다 |
| W1-3 | 로그인/로그아웃 | `GET/POST /login` · `POST /logout`. 실패는 균일 메시지(계정 존재 여부 비노출) |
| W1-4 | role 가드 | admin 구역 / user 구역 미들웨어. 부트스트랩 예외(W-D2) 포함 |
| W1-5 | 관리자 시드 | `provision_accounts.py` 확장 — 관리자 1 + 데모 유저 5(계좌 연결) |

**완료 기준**: 라우트 감사 테스트 — 유저 role의 쓰기 엔드포인트 0을
테스트가 강제(M10 원문 "라우트 감사로 검증"). 미로그인 → 로그인 리다이렉트.

## Phase W2 — 유저 포털 (read-only 3장)

| # | 태스크 | 상세 |
|---|---|---|
| W2-1 | 내 계좌 홈 | 총자산·수익률·계좌 곡선(`tb_account_equity_daily`)·보유 카드. **자기 계좌만** — 소유권은 세션 유저의 account_id로 쿼리 자체를 묶는다(뷰에서 거르지 않는다) |
| W2-2 | 거래 타임라인 | 체결 + 판단 조인 — **계보 FK가 이 화면의 존재 이유다**. "왜 샀나" = `tb_fill → tb_order → tb_strategist_signals`(서사·확신도) + `tb_critic_verdict`(반박문). 조인으로 답하고 기억으로 답하지 않는다 |
| W2-3 | 리스크 상태 | 국면(`tb_macro`)·risk_off 안내("오늘은 매수를 쉬었습니다")·브래킷 안내 |
| W2-4 | 벤치마크 수집 잡 | SPY 일봉 1종목 → `tb_benchmark_price`. 완료 후 W2-1에 SPY 대비 얹기 (W-D4) |

**완료 기준**: 다른 유저 계좌 URL 직접 접근 시 404(403 아님 — 존재 비노출).
화면 숫자와 원장 대조 테스트(결함 15·16의 교훈 — 화면은 원장보다 많이
세지도, 없는 분모를 지어내지도 않는다).

## Phase W3 — 관리자 구역

| # | 태스크 | 상세 |
|---|---|---|
| W3-1 | 관제실 이동 | 현 `GET /` → `/admin` 아래로. `/`는 로그인 상태별 리다이렉트(admin→관제실, user→내 계좌, 미로그인→로그인) |
| W3-2 | 계좌 총람 | 계좌별 수익률·보유·주문·지갑 상태 테이블 (M9-3) |
| W3-3 | 계좌 CRUD | 개설(유저 생성 포함)·성향 지정·paused/closed 전환 (M9-2). **관리자의 유일한 쓰기** — 주문 경로에는 여전히 손 없음 |
| W3-4 | 운영 콘솔 | 잡 체인 상태(관제실 기존) + **슬롯 재실행 버튼**(W-D3) + 방어선 발동 내역(`tb_order_plan` 보류 사유) |

**완료 기준**: user role로 `/admin/*` 접근 시 404. 슬롯 재실행이 완료 슬롯에
no-op 경고. 계좌 CRUD가 주문·체결 테이블에 닿지 않음을 테스트로.

## Phase W4 — 비용 대시보드 (M9-5 · open-items §3-1)

| # | 태스크 | 상세 |
|---|---|---|
| W4-1 | tb_llm_usage 배선 | `llm/provider.py`에 기록 훅 — 호출처·모델·토큰·추정비용. **두 번째 미배선 테이블이 닫힌다** |

| W4-3 | 대시보드 | 오늘/월·태스크별 집계 + 잔여 예산. openai 전환(R3) 결정의 데이터원 |

첫 데이터가 아니라 축적된 분포를 갖고 시작한다.

## 순서와 규모

W1 → W2 → W3 → W4 순. W2와 W3은 W1 위에서 독립적이라 병렬 가능하지만,
소유권 검증(W2)이 가드(W1-4)의 첫 실전 검증이므로 W2 먼저를 권장.

착수 규칙(playbook M9 원문): **첫 태스크 = 와이어프레임**. 아래 초안을
착수 세션에서 확정하고 구현에 들어간다.
