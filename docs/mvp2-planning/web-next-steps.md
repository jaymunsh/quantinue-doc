## 1. 웹 정리 — W1을 만들며 드러난 것

### 1-1. ⛔ 관제실의 "모의 원장" 패널이 **유령 계좌**를 본다 (막는 작업)

관제실에서 가장 큰 패널이 `quantinue-local-simulated` 계좌 하나를 보여준다.
그 계좌는 **1차 러너 시절의 유물**이고 지금은 아무도 안 쓴다:

| 계좌 | 현금 | 체결된 주문 |
|---|---:|---:|
| `quantinue-local-simulated` ← **화면이 보는 것** | $1,000,000.00 | **0** |
| TEST-AGGRESSIVE-01 | $20,539.52 | 4 |
| TEST-CONSERVATIVE-01 | $71,075.69 | 3 |
| DEMO-AGGRESSIVE-01 | $30,734.62 | 4 |
| DEMO-AGGRESSIVE-02 | $20,539.52 | 4 |
| DEMO-AGGRESSIVE-03 | $846.00 | 5 |
| DEMO-CONSERVATIVE-01 | $71,075.69 | 3 |
| DEMO-CONSERVATIVE-02 | $4,486.75 | 2 |
| TEST-DELISTED-01 | $20,592.72 | 8 |
| TEST-SELLGAP | $18,640.67 | 13 |

**즉 실제로 움직인 돈 전부가 화면에 안 보인다.** 총 46건이 체결됐는데 화면은
"보유 종목 없음 · 주문 0건 · 체결 0건"이라고 말한다. `GET /api/portfolio`도
같은 계좌를 본다.

이건 결함 15·16과 같은 계열이면서 **방향이 반대**다 — 화면이 많이 세는 게
아니라 **원장에 있는 것을 안 센다**. 시연에서 "그래서 실제로 뭘 샀나요"에
답할 수 없다.

- 원인: `store.simulated_portfolio(opening_cash)`가 계좌 하나를 가정한다
  (`postgres.py:102`, `self._account_identity`). M6-3에서 계좌 분할
  (`account_portfolio` / `account_portfolios`)을 이미 만들어 뒀고, playbook
  6-3 주석에 **"기존 `simulated_portfolio`는 화면이 쓰고 있어 존치 — 전환은
  M9"**라고 적혀 있다. 그 M9가 지금이다.
- 할 일: 관제실 원장 패널을 **계좌별**로 바꾼다(또는 계좌 총람 = W3-2로 흡수).
  `simulated_portfolio` 소비자가 0이 되면 함께 삭제.
- 규모: 읽기 계층 교체 + 템플릿. 스키마 변경 없음.

### 1-2. 구 토큰 별칭 블록 (의도적 과도기)

`dashboard.css`의 `:root`에 구 토큰 이름(`--card`·`--ink`·`--blue`…)이 새
정본 토큰의 **별칭**으로 남아 있다. 관제실 마크업을 한 줄도 안 고치고 팔레트
전체를 옮기려고 일부러 둔 것이다.

관제실을 시안 컴포넌트로 완전히 옮기면(§1-3과 함께) 이 블록이 사라진다.
**지금 지우면 안 된다** — 아직 그 이름들을 쓰는 규칙이 400줄 있다.

### 1-3. 마크업 계약이 테스트에 **문자열로** 박혀 있다

`tests/test_pipeline_dashboard.py:98`이 잡 목록을 이렇게 찾는다:

```python
opening = body.index('<ol class="job-chain-list">')
```

CSS가 페이지에 인라인되기 때문에 생긴 방식인데, 부작용이 둘 있다:

1. **클래스를 덧붙일 수 없다.** `class="job-chain-list status-list"`로 바꾸면
   화면은 멀쩡한데 테스트가 실패한다. 그래서 지금은 스타일을 `.job-chain-list`
   선택자에 따로 주고 있다.
2. **CSS 주석에 그 문자열을 적으면 안 된다.** 실제로 밟았다 — 계약을 설명하는
   주석을 CSS에 적었더니 슬라이스가 본문보다 주석을 먼저 찾아 엉뚱한 구간을
   잘랐다(`9ba8e74`).

→ 개선안: 마커를 `id` 하나(`id="job-chain"`)로 바꾸고 테스트가 그걸 잡게 한다.
클래스는 자유로워지고 주석 충돌도 사라진다. **테스트 삭제 규칙대로 고정하던
코드와 같은 커밋에서 교체한다.**

### 1-4. 모바일에서 사이드바가 드로어가 아니라 눕는다

시안은 720px 이하에서 사이드바를 **드로어**로 밀어 넣고 `app.js`가 토글한다.
우리는 자바스크립트를 안 들이려고 **위로 눕히는 방식**을 골랐다.

내비 항목이 5개인 지금은 충분하지만, W3에서 관리자 화면이 늘면 위쪽 띠가
두꺼워진다. 그때 드로어를 다시 볼 것 — 그러면 이 앱의 **첫 자바스크립트**가
된다(지금은 관제실 표 포커스 처리 11줄이 전부다).

### 1-5. 로그인만 셸 밖에 있다

`login.html`은 `base.html`을 상속하지 않는다(상단바에 로그아웃·모드 칩이
앉을 자리가 없고, 시안의 2단 구성이 화면 전체를 소유한다). 의도한 것이지만
**CSS 토큰이 갈라지지 않게 주의**해야 한다 — 로그인만 다른 색이 되면 그건
사고다.

### 1-6. 쓰지 않는 CSS 규칙 3개

시안에서 옮겨오며 소비자 없이 들어온 것: `.metric__delta--good` ·
`.metric__delta--bad` · `.grid-2--balanced`. 수익률 표시(W2-1)가 붙으면
앞의 둘은 바로 소비자가 생긴다. **W2-1까지 안 붙으면 지운다.**

## 5. 웹이 아닌 관찰 — 잡 실행 (이번에 눈에 띈 것)

화면을 만들다 원장에서 발견한 것들이다. **아직 원인을 확인하지 않았으므로
추정을 사실로 적지 않는다.**

### 5-1. 07-21 슬롯에 잡이 3개만 돌았다

```
screening            succeeded  0 picks from 0 ranked for 2026-07-20
analysis:aggressive  succeeded  0 analysed
analysis:conservative succeeded 0 analysed
```

`daily_bars`는 주기가 1일이고 마지막 성공이 07-20이라 **07-21에 돌았어야
한다.** 안 돌았고, 그래서 스크리닝이 볼 봉이 0이고, 그래서 픽이 0이고,
그래서 분석도 0이다. 체인의 앞이 비면 뒤가 조용히 0을 보고하는 형태다.

⚠️ 이 세션에 앱을 여러 번 재시작했으므로 **그 영향인지 실제 결함인지
가려야 한다.** 재시작 없이 하루를 돌려보는 것이 첫 확인이다(open-items §4-3
"무인 완주"와 같은 항목).

### 5-2. 잡 소요시간이 실행 시간이 아닐 수 있다

07-20 슬롯 실측:

| 잡 | started_at | finished_at | 표시 |
|---|---|---|---|
| news | 07-19 15:23 | 07-20 01:11 | **35,290s (9.8시간)** |
| daily_bars | 07-19 23:43 | 07-20 01:11 | **5,270s** |
| analysis:aggressive | 07-19 18:35 | 07-19 18:41 | 368s |

news가 실제로 9.8시간 걸렸을 리 없다(실측은 14.5초였다). 같은 슬롯을 다시
돌릴 때 `started_at`이 **첫 시도 값으로 남는 것**으로 보이는데, 확인은
안 했다. 사실이면 관제실의 소요시간 표시가 **재실행된 잡에서 거짓말을 한다.**

→ 확인 방법: 슬롯을 하나 지우고 두 번 돌려 `started_at`이 갱신되는지 본다.
