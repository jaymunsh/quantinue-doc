## R1. 2차의 정의 — "AI 펀드 서비스" (2026-07-18 확정)

**2차(최종) = 전체 자동화 + 관리자/유저 2면 서비스.**

1. **무인 페이퍼 운영이 토대** — 사람 손 없이 하루 사이클 자동 완주(스케줄 배선) + 판단 방어선 실작동 + 회고 루프. 전체 자동화가 2차의 포인트.
2. **관리자 화면 = 펀드매니저 + 시스템 운영 (ERP 느낌)**
   - 계좌 CRUD (1유저=1계좌): 개설·성향 지정·일시정지·해지
   - 계좌 총람: 계좌별 수익률·보유·주문 현황
   - 운영 콘솔: 파이프라인 실행·라이브 진행·방어선 발동 내역 (기존 Control Room 계승·확장)
   - **API 사용량·남은 AI 비용(예산) 대시보드**
   - **자동매매 외적 시스템 유지보수 기능** (상세는 설계 시: 로그·잡 상태·데이터 정합 점검·설정 관리 등)
   - 리포트 발행 관리
   - **유저 계정 생성도 관리자가 수행** (셀프 가입 없음) — 인증은 OTP 등 방식을 설계 시 결정
3. **유저 화면 = 조회 전용 (read-only)**
   - 내 계좌 홈 (총자산·수익률 곡선·보유 종목)
   - 거래 타임라인 (매수/매도 + 이유 — DB의 reason/summary를 사람 언어 피드로)
   - 매니저 리포트 (Titan 스타일: 매크로 regime·오늘 산/안 산 이유·주간 요약)
   - 투명성 리포트 (T+5 회고 결과 공개 — 차별점)
   - 리스크 상태 ("오늘은 risk-off라 매수 보류")
   - 유저의 유일한 쓰기 = 없음 (성향도 관리자가 지정)
4. **투자 성향 = 2종: 공격형(aggressive) + 안전형** — 기존 3종 구상에서 축소. 1차는 공격형 단일이었음. (안전형의 영문 ENUM 명명은 컨벤션 규칙 2에 따라 확정 필요 — `conservative` 제안)
5. **아키텍처 원칙: 리서치는 공용, 계좌는 구독** — 01~08 신호 생성은 사이클당 1회, 계좌는 성향별 게이트·사이징(09~10)만 계좌별 루프. LLM 비용이 계좌 수와 무관하게 고정.
6. **API 호출 횟수(주기)는 비용 현실을 반영해 조정** — 구체 수치는 R5(LLM·비용 라운드)에서 확정.
7. **실계좌 전환은 범위 외** — 페이퍼 전용 유지. 멀티 계좌 확대(5개+)는 2차 안정화 후 확장 옵션.

**근거**: config phase_2(accounts 5)·tb_order UNIQUE(account_id, signal_id)·tb_account 멀티 계좌 스키마 등 팀의 원 구상이 이미 멀티 계좌 지향. 유저 read-only는 "LLM 출력이 검증 없이 돈에 닿는 경로 0개" 보안 서사 유지 + 구현량 통제.

## R2. 스케줄 엔진·cycle_ts 멱등 (2026-07-18 확정)

1. **스케줄 엔진 = 앱 내장 (A안)** — FastAPI lifespan 백그라운드 루프가 기존 `DueRoleScheduler`(현재 미배선 seam)를 주기 호출. 24시간 데몬. Celery·외부 cron 비채택 (과함/인프라 의존).
2. **cycle_ts = 역할별 슬롯 경계로 내림** — 실행 시각이 아니라 "어느 칸의 실행인가" 라벨 (뉴스 30분 칸·전략 120분 칸). **수동 실행(관리자 버튼)도 같은 규칙** → 자동+수동 혼재 시에도 UNIQUE(ticker, cycle_ts) 멱등 유지. 현행 `now()` 분내림(main.py:52)은 멱등 파손 버그로 수정 대상 1순위.
3. **슬롯 마스터 테이블 없음 (1차 #19 결정 유지·재확인)** — 슬롯은 계산값(시각÷주기 내림). 실행 기록은 이미 있는 `pipeline_runs`/`pipeline_stage_attempts`가 담당 (1차 설계의 tb_job_run 제안이 코드에서 이 이름으로 선구현됨 — v4.0 문서에 정식 등재).
4. **역할별 실행 창 (뉴욕 시간 정의, 실행 시 변환 — 서머타임 안전)**: 01~03 장전 1회 · 04 장전+장중 · 05 확장시간(장후 공시 포함) · 06 장전~장중 · **07~10 개장 시간만(주문 경로)** · 11 장 마감 후. 정본 #sched 하루 타임라인 계승.
5. **배포**: 로컬 구축·부분 테스트 → AWS 이전. 소형 인스턴스(t3.small급)로 충분(API 호출 위주 워크로드), 기존 Dockerfile 활용. **⚠️ AWS는 최후·선택 사항(R9 재확인)** — 로컬 상시 구동(절전 방지 caffeinate)으로 대체 가능하며, 2차 "완료" 기준에서 제외한다. 완료 기준 = **로컬에서 무인 하루 사이클 완주 + 화면 동작**.

**뒤집기 조건**: 앱 프로세스 다중화(수평 확장)가 필요해지면 그때 외부 스케줄러 재논의 — 페이퍼 운영 규모에선 불필요.

## R8. 스크리닝 깔때기 — 원설계 복원 + 실무 보정 (2026-07-18 확정)

**배경**: 정본 원설계 = 주1회 시총 top 2,000(tb_universe 앨범) → 매일 "오늘의 50"(5버킷). 그러나 1차 코드는 "스크리너 응답 앞 50(시총 정렬도 아님) → 기술 20 → 픽 20"으로 임의 축소 — 촉매 전략의 감시망이 50종목이면 재료 대부분을 놓침. **축소는 버그로 판정, 원설계 복원.**

**확정 깔때기: 16,000 → 2,000(주1회) → 50(매일) → 20+보유(LLM 심층)**

1. **주 1회**: 시총 상위 2,000 → tb_universe append (원설계 — 생존편향 방지 PK(as_of_date, ticker) 유지)
2. **매일 장전**: 2,000 전종목 일봉 벌크 수집(Alpaca 무료 IEX·멀티심볼 bars — 분당 200req 내 수십 콜로 충분, 검증 완료 07-18) → **하드 필터 신설(실무 보정): 주가 ≥ $5 · 20일 평균 거래대금 ≥ $20M** ⚙️ → 02 지표 → 03 버킷 스코어링 → **오늘의 50** (tb_daily_pick)
3. **장중**: 공시·뉴스 수집 = 오늘의 50 ∪ 보유 / **LLM 심층(05~08) = 점수 상위 20 ∪ 보유** — 코드의 "20"은 LLM 비용 캡으로 재해석·존치
4. 데이터 소스: 유니버스 목록 = NASDAQ screener API(무료) · 일봉 = Alpaca Market Data 무료 티어(주문과 동일 키)

## R9. 수동 운영·장애 복구 (2026-07-18 · Phase 3 중 추가)

**배경**: 앱 내장 스케줄러(R2)는 PC/서버가 켜져 있어야 돈다. PC 24시간 구동이 보장 안 될 수 있고 스케줄 실행이 실패할 수도 있으므로, 수동 실행을 정식 기능으로 둔다(= 장애 복구 수단 겸용).

1. **수동+자동 무충돌**: 두 경로 모두 R2 슬롯 함수로 cycle_ts 계산 → 같은 슬롯은 `UNIQUE(ticker, cycle_ts)`로 멱등. + `(component, cycle_ts)` 실행 **claim 락**으로 동시 중복 실행 차단(1차 기존 idempotent claim 확장). 근접 발화해도 하나만 실행, 나머지 no-op.
2. **catch-up(따라잡기)**: 기동 시 또는 관리자 버튼으로 `DueRoleScheduler`가 "오늘 아직 기록 없는 due 역할"을 현재 데이터로 실행. **한계: 과거 슬롯 소급 백필 아님**(장이 지남) — 하루 통째 놓치면 그날은 포지션 없음(허용).
3. **관리자 수동 컨트롤(M9)**: 지금 사이클 실행 · 특정 역할만 실행 · 밀린 슬롯 따라잡기 · 실패 단계 재실행(체크포인트 재개) · 리뷰 재채점. 각 버튼은 대상 슬롯 표시 + 완료된 슬롯이면 멱등 no-op 경고.
4. **장애 복구 연동**: 스케줄 실패 → 텔레그램 알림(R7-3) → 관리자 재실행(체크포인트 재개, 중복 없음).

**영향**: M1(스케줄러에 catch-up 배선) + M9(수동 컨트롤 UI). 대부분 1차 자산(claim·checkpoint·retry·DueRoleScheduler·`POST /runs`) 재활용.

## R10. 07·08 계보 컬럼 추가 (2026-07-18 · Phase 3 중 추가)

**배경**: 감사 결과 계보(provenance) 10컬럼이 05·06 수집 계열(공시·뉴스 원본·집계)과 tb_review_price_snapshots에만 구현됨. 그런데 **tb_strategist_signals(07)·tb_critic_verdict(08)도 LLM 산출**인데 계보가 없어, "판단 자체를 만든 모델·프롬프트 버전"이 그 행에 없다(간접 추적만 — src_*_at·signal_id·pipeline_runs).

- **결정: 07·08에도 계보 10컬럼 추가 (M2 스키마 작업 포함).** 1차 #16(재현 계약)이 "2차 항목"으로 미뤄둔 것의 이행. 회고(R6)가 판단 품질을 평가하는데 판단의 레시피(프롬프트·모델·policy 버전)가 행에 없으면 "전략 탓 vs 프롬프트 탓" 귀책 구분이 불가 → 회고 분석 반쪽.
- **비용 무시 가능**: 계보는 짧은 문자열/해시, 행당 ~200~300B, 인덱스 없음, 판단 로직이 읽지 않음(사후 조회 전용). 05·06에 이미 검증된 패턴을 07·08에 복사 — 개발 비용도 작음. (진짜 용량 이슈는 원장 성장 = #9, 별개 이연 안건.)
- 반영: 데이터 모델 스키마 명세(07·08에 계보 행 +M2 배지)·계보 설명 카드·ENUM 사전 model_provider 쓰이는 곳·M2 변경표.
