## 2. 진단 — 무엇이 표류했나

- MVP-1은 Alpaca 페이퍼 "무장 대기"까지 갔다가, D1·D2에서 MockBroker 시뮬이
  "최종 상태"로 확정됐다. → **C1과 정렬** (표류 아님. 오히려 R1 실브로커
  로드맵이 불필요해졌다).
- MVP-2 재설계에서 실행 주기가 "뉴욕 세션일마다 1회"로 굳었고, 이후 세션들이
  이를 "확정, 되묻지 말 것"으로 재생산했다. → **C2와 어긋남. 유일한 진짜 격차.**
- LLM 제안 + 크리틱 반박 + 예산 상한 구조. → **C3과 정렬** (자산).

즉 **부수고 다시 지을 것은 없다.** 판단 층은 그대로 두고, 가격을 추적하며
방어선을 장중에 발동시키는 **감시 층 하나를 얹는다**.

## 5. 목표 설계 — 두 층 구조

실시간성의 가격은 층마다 다르다. 그래서 최적점도 층별로 따로 잡는다.
**"판단은 하루 1회"가 아니다** — 하루 1회인 것은 "새 종목 발굴"뿐이고,
이미 고른 종목들에 대한 판단·매매는 장중에 계속 갱신된다:

| 층 | 주기 | 근거 |
|---|---|---|
| 세계 선정 — 유니버스·스크리닝·픽 선발 | 세션일 1회 (변경 없음) | 재료(일봉·공시)가 하루 단위로만 갱신 — 더 돌려도 정보가 안 늚 |
| 판단 갱신 — 보유+당일 픽(~40종목) 재평가 | **장중 정기 스윕 2~3회 + 사건 구동 즉시** | 스윕당 ≈ 일일 분석 1회 비용(~$0.05) → 하루 ~$0.2 수준, 예산 $3 내 |
| 매매 집행 | 판단이 바뀌는 즉시 — 매도든 **매수든** | 스윕·재판단의 매수 판단은 갭 가드 통과 시 그 자리에서 체결 |
| 방어 — 손절·익절 브래킷 | **1분 폴링** → 추후 초 단위(스트리밍, M7) | 감시 대상 ≤40종목 = 배치 1콜/분 = Alpaca 한도(200/분)의 0.5%. 사실상 공짜 |

```
[일일 층]   기존 잡 체인 13종 — 뉴욕 세션일마다 1회 (변경 없음)
            재료: 일봉·공시·뉴스·금리 → 스크리닝·픽 선발 → LLM 판단
            → 사이징 → 매수. 산출: 포지션 + 손절·익절 브래킷 + 오늘의 픽

[감시 층]   price_watch 루프 — 정규장 중 1분마다 (신규, M1~M3)
            재료: 보유·당일 픽 종목의 현재가 (~10~40종목, 배치 1콜)
            행동: ① 브래킷(손절·익절) 도달 → MockBroker.close 즉시 체결
                  → tb_order(closes_order_id)·tb_fill 기록 → 🛡 텔레그램
                  ② 급변·뉴스 트리거 감지 → 재판단 큐에 넣는다

[재판단 층] 장중 LLM 재평가 (신규, M4~M5)
            호출: ⓐ 사건 구동 — 감시 층 트리거 즉시 (종목당 쿨다운)
                  ⓑ 정기 스윕 — 뉴욕 시각 2~3회, 보유+당일 픽 전체
            행동: 매도 판단 → 즉시 청산 · 매수 판단 → 갭 가드 통과 시
                  즉시 매수 · 판단마다 크리틱 반박 + 예산 가드 통과 필수
```

동작 시나리오 (하루):

1. KST 13:00 — 일일 층이 오늘 슬롯을 돈다(현행과 동일). 매수 체결과 함께
   손절 −15%/익절 +20% 다리가 MockBroker에 상주로 걸린다.
2. 뉴욕 정규장(KST 22:30~05:00, 서머타임 기준) —
   감시 층이 1분마다 현재가를 조회한다. 손절가에 닿으면 **1분 안에**
   청산되고 알림이 온다. 급변(±N%)이나 속보가 감지되면 그 종목만
   **즉시 LLM 재판단**이 돌고, 판단이 뒤집히면 청산·신규 매수까지
   그 자리에서 이어진다. 별개로 정기 스윕이 개장 직후·중반·마감 전에
   보유+픽 전체를 재평가한다.
3. 장 마감 후 — 감시·재판단 층은 쉰다(세션 게이트). 다음 세션일 13:00에
   일일 층이 다시 돈다.

원칙: **LLM은 1분 루프 안에 없다.** 1분 루프는 규칙(브래킷·트리거 판정)만
돌리고, LLM은 **사건과 스케줄된 스윕**이 깨운다. 판단(뇌)과 방어(반사신경)의
분리 — 이 구조라야 실시간성(1분 감시·즉시 재판단)과 비용(~$0.2~0.5/일 추가,
예산 $3 내)이 동시에 성립한다.

### 개발 전후 — 같은 상황이 어떻게 달라지나

| 상황 | 현재 (M1 이전) | 완성 후 (M5 이후) |
|---|---|---|
| 보유 종목이 장중 −16% 급락 | 아무 일도 안 일어남. **다음날 13:00** exits가 일봉을 보고 손절 — 하락을 하루 더 안고 감 | **1분 안에** 손절 체결 + 🛡 알림. 급변 트리거로 LLM 재판단도 돌아 사유가 원장에 남음 |
| 보유 종목에 장중 악재 속보 | 다음날 슬롯에서야 뉴스가 판단에 반영 | 트리거 → 그 종목 즉시 재판단 → 판단 반전 시 그 자리에서 청산 |
| 익절가(+20%) 장중 도달 후 반락 | 다음날 일봉으로 평가 — 고점을 놓치고 반락분을 반납할 수 있음 | 도달 시점(1분 내)에 익절 체결 |
| 아침에 못 산 픽이 장중에 조건 충족 | 기회 없음 — 매수는 13:00 일괄뿐 | 스윕·재판단에서 매수 판단 시 갭 가드 통과하면 장중 매수 |

## 6. 코드 변경 지도

### 6-1. 신규: `src/quantinue/orchestration/watch_runner.py`
`JobRunner`와 대칭인 얇은 루프. `run_forever`로 `main.py` lifespan에 함께
띄운다. 매 틱: ① 정규장인지 판정(뉴욕 캘린더 + 장 시간) ② 감시 대상 산출
(보유 전량 + 당일 planned 주문 종목) ③ 시세 배치 조회 ④ 브래킷 평가 →
청산 실행. 실패는 격리하고 텔레그램으로 알린다(러너의 실패 처리 패턴을 따라).
**`tb_job_run`에 틱을 기록하지 않는다** — §6-5.

### 6-2. 수정: `orchestration/policy.py` + `config/pipeline.yaml`
`WatchConfig` 모델 신설 (frozen, extra="forbid" — 기존 관례):

```yaml
mvp2:
  watch:
    enabled: false          # 기본 꺼짐 — 관측 인스턴스는 명시적으로 켠다
    interval_minutes: 1     # 배치 1콜/분 = Alpaca 한도(200/분)의 0.5%
    session: regular        # 정규장만. 프리·애프터는 명시적 확장 전까지 없음
    rejudge:                # M4~M5: 장중 LLM 재판단
      enabled: false
      move_trigger_pct: 0.05   # 기준가(직전 종가) 대비 이 변동이면 트리거
      cooldown_minutes: 30     # 같은 종목 재판단 최소 간격 — 진동 방지
      sweep_times_ny: ["10:00", "12:45", "15:15"]  # 정기 스윕(뉴욕 시각)
      sell_budget_reserve_ratio: 0.2  # 매도성 재판단 전용 예산 예약 (§6-5)
```

주기·시간대 값은 코드 리터럴 금지 규칙 그대로 이 블록이 소유한다.

### 6-3. 신규: `market_data/alpaca_quotes.py` (또는 alpaca_bars 확장)
감시 대상 종목의 최근 체결가 배치 조회. `symbols_per_request`(200) 안에서
틱당 1콜이면 충분하다 — Alpaca 분당 한도 200 대비 10분당 1콜은 무시 가능.
테스트·mock 모드용으로 `fixture.py`에 장중 시세 픽스처를 추가한다.
(C1의 "로컬 데이터 기반"은 **체결**이 로컬이라는 뜻이다. 시세는 현행대로
Alpaca 실물을 쓴다 — 이미 일봉·뉴스가 그렇다.)

초 단위로 더 조이는 업그레이드 경로는 Alpaca 무료 실시간 스트리밍(웹소켓,
IEX 피드)이다. **2026-07-22 실제 계정으로 30종목 구독 성공, 31번째 종목은
`symbol limit exceeded`를 확인했다.** 따라서 감시 대상 ~40종목 전체를
스트리밍으로 바꾸지 않는다. 보유 종목(계좌당 최대 10, 전체 최대 30)만
스트리밍하고, 오늘의 픽과 30종목 초과분은 기존 1분 배치 폴링을 유지한다.
스트림이 끊겨도 폴링은 계속 돌아 최종 방어선이 사라지지 않는다. 스트리밍은
`watch.stream.enabled=false`가 기본이라 운영자가 명시적으로 켜기 전에는
연결도 만들지 않는다.

### 6-4. 수정: `roles/exits/` — 방어선 ①층 소유권 이관
`decide_exit`(contracts.py:126)의 4층 중 **①브래킷 평가를 독립 함수로 분리**해
감시 층이 현재가로 호출한다. 일일 `ExitJob`은 ②명제 붕괴 ③약세 판단
④시간 청산만 남긴다(셋 다 재료가 일 단위라 일 1회가 맞다). 같은 브래킷의
이중 발동은 `tb_order.idempotency_key`가 막는다(일일 잡과 감시 층이 같은
청산을 두 번 시도해도 한 번만 체결). 분리하면서 이 파일의 주석 중 폐기된
R1을 근거로 인용하는 부분("나중에 페이퍼로 전환할 때" 등)을 헌장 C1 기준으로
고쳐 쓴다 — 낡은 근거를 남기면 다음 독자가 옛 방향으로 오독한다.

### 6-5. 신규: 장중 재판단 경로 (M4~M5)
기존 `roles/analysis/`의 제안+크리틱 계약을 **단일 종목 단위로 재사용**하는
엔트리를 만든다 — 전체 슬롯 분석과 프롬프트·계약을 공유한다(비슷한 코드를
새로 쓰면 유령이다). 트리거 큐는 감시 러너 내부 메모리로 충분하다(재기동
시 소실 허용 — 다음 틱이 같은 조건을 다시 감지한다). 재판단 산출은 기존과
동일하게 `tb_strategist_signals`/`tb_critic_verdict`에 기록하되, 일일 판단과
구분 가능해야 한다(기존 `cycle_ts`·`source` 컬럼으로 충분한지 먼저 확인 —
스키마 확장이 필요하면 4곳 미러 + 멱등 2회).

**예산-매도 상호작용 (규정)**: "예산 초과 = skip"을 장중 재판단에 그대로
적용하면 안 된다 — 매수 skip은 안 사는 쪽이라 안전하지만, **매도성 재판단
skip은 위험 보유를 유지하는 쪽**이고, 급변장일수록 트리거가 몰려 예산이
먼저 마르는 역설이 생긴다. 그래서 일 예산의 일부를 매도성 재판단 전용으로
예약한다(`sell_budget_reserve_ratio: 0.2` 확정 — pipeline.yaml 소유):
잔여 예산이 예약분 이하로 내려가면 **매수성 재판단만 skip**하고, 보유 종목
대상 재판단은 예약분에서 계속 돈다. 예약분마저 소진되면 브래킷(1분 루프,
LLM 무관)이 최후 방어로 남는다.

**장중 매수의 사이징 (M5 규정)**: 사이징·현금 가드·포지션 캡·일일 신규
상한은 **배분 잡(`roles/allocation/`)의 기존 계약을 단일 종목으로 재사용**
한다 — 감시 층에 새 사이징 로직을 쓰면 유령이다. "갭 가드"는
`gates.premarket_gap_max`(0.03) + `gates.gap_guard_open_minutes`(30) 창을
그대로 적용한다(둘 다 pipeline.yaml `gates` 소유, 기존 개장 가드와 동일
개념 — 새 개념 아님).

### 6-6. 원장: `tb_job_run`은 건드리지 않는다
PK `(job_name, slot_date)`의 "잡 하나=하루 한 번" 불변식은 화면·재시도·
알림이 모두 기대는 계약이다. 감시 틱(하루 ~39회)을 여기 넣으면 관제실
의미가 깨진다. 감시 층의 산출은 **체결 원장(tb_order/tb_fill)에만** 남기고,
상태 가시화가 필요해지면 경량 하트비트(예: tb_watch_tick 1행 upsert)를
그때 추가한다 — 스키마 추가 시 "4곳 미러 + 멱등 2회" 규칙 적용.

### 6-7. 수정: `orchestration/job_factory.py`, `main.py`
감시 러너 조립 함수(`build_watch_runner`) 추가, lifespan에서
`task_group.start_soon` 한 줄 추가. 기존 잡 등록 순서는 변경 없음.
