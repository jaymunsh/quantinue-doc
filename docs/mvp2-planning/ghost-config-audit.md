# 유령 설정·컬럼 감사 (2026-07-19)

> M4에서 같은 패턴의 결함이 반복해서 나와(`late_entry_max`·`peak_importance`·`signal_consensus` 전부 "선언됐는데 소비자 없음") `mvp2` 블록과 주요 테이블을 **전수 조사**한 기록.
> **원칙**: 선언된 것은 배선하거나 삭제한다. config에 값이 있는데 아무 효과가 없는 상태가 쌓이면 시스템 신뢰가 죽고, **M7 학습 루프가 이걸 진짜 데이터로 착각한다.**

## 왜 이게 위험한가

세 가지가 동시에 일어난다:

1. **문서와 코드가 갈라진다** — 설계 문서·playbook은 "한도 10개, 손실 4% 제한"이라 적혀 있는데 코드는 아무것도 안 막는다. 리뷰어도, 미래의 나도 문서를 믿는다.
2. **조정이 무력하다** — yaml을 고쳐도 아무 일도 일어나지 않는다. "문턱은 config가 소유한다"는 규칙이 거짓이 된다.
3. **학습이 오염된다** — M7은 DB 값을 진실로 읽는다. `signal_consensus`가 늘 0이면 "합의는 성과와 무관하다"는 교훈을 학습한다.

## 미해소 — 우선순위 순

### 1. LLM 예산 상한이 전혀 강제되지 않는다

`budget.daily_llm_usd`(현재 $3) 소비자 **0**. 뒷받침 테이블 `tb_llm_usage`도 참조 **0** — 쓰는 곳도 읽는 곳도 없다.
→ **M8-2·8-3의 실체**. 그때까지 LLM 비용에 상한이 없다는 뜻이고, 로컬 MLX를 쓰는 동안은 금전 리스크가 없지만 클라우드 모델로 전환하는 순간 무방비다.

### 2. 성향별 리스크 한도 — ✅ 3/5 해소 (M6-2, 2026-07-19), 2종 잔존

| 필드 | 상태 |
|---|---|
| `max_weight`(0.20/0.10) | ✅ **해소** — `POSITION_CAP_FRACTION` 0.25 리터럴을 대체. 같은 개념이 리터럴과 유령 config로 이중 존재하던 것을 하나로 |
| `max_positions`(10/5) | ✅ **해소** — 보유 종목 수 한도 도달 시 신규 진입 차단 |
| `min_cash_ratio`(0.10/0.30) | ✅ **해소** — 매수 뒤에도 현금 바닥 유지 |
| `daily_loss_limit`(0.04/0.02) | ⏳ **잔존 — M5 이후**(아래) |
| `risk_off_action`(penalty/no_new_buys) | ⏳ **잔존 — 별도 항목**(아래) |

함께 해소된 것: 자본이 계좌에서 오게 됐다(그 전엔 앱 전역 노출 상한이 equity였다). 상세는 playbook M6-2.

#### ⏳ `daily_loss_limit` — 왜 미루었고, 무엇이 갖춰지면 되는가

**지금 구현하면 절대 발동하지 않는다.** 두 가지가 동시에 막고 있다:

1. **실현손실이 구조적으로 0** — 매도 경로가 없다(M5 미구현). 판 적이 없으니 실현된 손실이 없다.
2. **미실현손실을 잴 수 없다** — `tb_account.equity`는 체결 시 **갱신되지 않는다**. 매수하면 `cash`와 `buying_power`만 차감되고(`postgres_accounting.py:76-87`) equity는 최초 자본에 고정된다. 보유 포지션의 시가평가가 어디에도 없다.

그래서 지금 넣으면 **"손실 0 vs 한도 4%"를 영원히 비교하는 유령**이 하나 더 생긴다 — 이 문서가 경고하는 바로 그 패턴이다.

**착수 전제(전부 충족되어야 함)**:
- [ ] **M5 매도 경로** — 실현손익이 발생하기 시작
- [ ] **보유 포지션 시가평가** — 미실현손익 산출. M7-4가 도입할 일별 가격 수집(`tb_benchmark_price` 계열)을 재사용 가능
- [ ] **당일 시작 equity 스냅샷** — ⏳ playbook M6-2에 원래 있던 항목. 설계안: `tb_account_equity_daily(account_id, trade_date, opening_equity)` 1행/계좌/일. **소비자(이 서킷)와 같은 커밋에 만들 것** — 먼저 만들면 또 유령이 된다
- [ ] 발동 시 **당일 신규 매수 중단 플래그 + 익영업일 자동 해제**(M1 캘린더의 `add_business_days`)

**완료 판정**: playbook M6 완료 기준의 **E2E-3**(일손실 초과 → skip · 익영업일 해제). 이 E2E가 green이 되기 전에는 `daily_loss_limit`을 "적용됨"으로 기록하지 말 것.

#### ⏳ `risk_off_action` — 매크로 위험 국면의 행동 분기

`penalty`(확신도 감점) vs `no_new_buys`(신규 매수 중단). 현재 소비자 **0** — 매크로 감점표(`gates.macro_penalty_table`)는 성향과 무관하게 **모든 계좌에 penalty 방식으로만** 적용된다. 즉 안전형 계좌도 위험 국면에서 매수를 계속한다.

**착수 조건**: 전제 없음 — role_07의 `vote_conviction`에서 프로필을 보고 분기하면 된다. **M6-1(계좌 구독 루프)에서 계좌별 프로필이 role_07까지 닿을 때 함께 처리하는 것이 자연스럽다**(현재 role_07은 단일 기본 프로필만 받는다).

### 3. `conservative` 프로필은 도달 불가

`factory.py`가 `DEFAULT_PROFILE_NAME = "aggressive"`로 하드코딩하고, yaml의 `mvp.policy_profile`은 읽는 곳이 없다. 안전 성향 블록은 통째로 유령.
→ **M6-1 계좌 구독 루프**가 해소해야 할 항목.

### 4. `screening.llm_depth`(20) — 유령이지만 **위험하지 않다** (초판 오류 정정)

**초판에서 "픽 50개 전부가 05~08을 탄다 · M3 기록과 불일치"라고 적었으나 사실이 아니다.** 확인 결과 **다종목 팬아웃이 애초에 존재하지 않는다** — 모든 런이 단일 종목이고(`main.py:57`·`:98`), 05~08은 `context.request.ticker` 하나만 분석한다. M3 기록의 "LLM 심층은 요청 종목 한정(보유 합류 훅은 M5)"이 정확한 서술이며, "LLM 호출 종목 ≤ 20"은 자동으로 충족된다. 비용·시간 문제 없음.

`llm_depth`가 소비자를 갖지 못한 진짜 이유는 **자를 대상이 아직 없어서**다. **M5-1**(분석 대상 ∪ 보유)이 팬아웃을 만드는 순간 이 키의 소비자가 생긴다.

> 📌 **이 항목이 감사 자체의 교훈이다.** "소비자 없음"은 grep으로 기계적으로 판정되지만, **그게 왜 위험한지는 기계적으로 판정되지 않는다.** 유령 목록을 읽을 때 각 항목의 결과를 코드로 다시 확인할 것 — 이 감사의 초판이 바로 그걸 건너뛰고 틀렸다.

### 5. `exits.time_exit_bdays`(10) 미적용

시간 기반 청산 로직 자체가 없다. 포지션에 보유 기한이 없다.
→ **M5-4 청산 3층**의 ② 항목.

### 6. 차단 사유가 계산되고 표시되지만 저장되지 않는다

`skipped_rules`(크리틱)·`hard_block_reason`(공시·뉴스) 모두 도메인에서 생성되고 화면에 렌더되는데 **insert에 빠져 있다** → 왜 차단됐는지 DB로 사후 추적 불가.

### 7. 기타 유령 컬럼

- `tb_strategist_signals`: `src_disclosure_at`·`src_news_at`·`src_macro_at`(출처 FK가 항상 NULL) · `bull_case`·`key_risk`·`risk_rebuttal`·`counter_scenarios`·`persona_notes` · 계보 10필드
- `tb_strategist_signals` **퇴화 값**: `sizing_hint`는 늘 `{}`, `current_price`/`day_high`/`day_low`/`close_prev`/`high_52w`/`low_52w`가 **전부 `decision_close` 하나로 채워짐**, `volume`/`turnover`는 늘 0 — 값이 있는 것처럼 보이지만 정보가 없다
- `tb_disclosure_signal`: `importance`·`risk_score`·`sentiment_score`·`disclosure_count`·`top_evidence`·`summary` 미기록(원본 `tb_disclosure`에는 있는데 정규화 행에는 없음 — `tb_news_signal`과 비대칭)
- `tb_news_signal`: `disclosure_ref`·`hard_block_reason`·`top_evidence` 미기록
- `tb_order.updated_at`: **어떤 UPDATE도 쓰지 않고 트리거도 없다** → 상태가 여러 번 바뀌어도 `created_at == updated_at`

## 이 감사 자체에서 배운 것 (2026-07-19, 정정 3회)

하루에 세 번 틀렸다 — `llm_depth`의 결과 · W0-7 경고의 성격 · 위 125% 산수. 공통점이 뚜렷하다:

**grep으로 "소비자 없음"을 확인한 뒤 그 결과를 추정했을 때 세 번 다 틀렸고, 실제 코드 경로를 끝까지 읽었을 때 맞았다.**

- `llm_depth` — 소비자가 없는 건 맞지만 팬아웃 자체가 없어 위험이 아니었다
- 리스크 한도 — 미적용은 맞지만 다른 한도 둘이 실제로 작동 중이었고, 문제는 그 **값**이었다
- 125% — 예약 함수 안의 한도 **둘 중 하나만** 보고 계산했다

**규칙**: 유령 목록의 각 항목은 "왜 위험한가"를 **호출 경로 끝까지 읽어** 확인하기 전에는 보고하지 않는다. 정적 검색은 후보를 찾을 뿐 결론을 주지 않는다.

## 재발 방지

새 config 키·DB 컬럼을 추가할 때 **같은 커밋에 소비자와 테스트를 넣는다.** 소비자가 다음 마일스톤이라면 키를 지금 만들지 않는다 — 설계 문서에만 적어두고, 배선하는 마일스톤에서 키를 만든다.
